P0（必须修复，阻塞开发）

P0开发1. 账号注销身份验证方式未定义（OAuth 用户无密码）
位置：PRD 第 4.6 节 账号注销
接口 /api/auth/delete-account 标注"需身份验证"，但未说明验证方式。邮箱注册用户可以输入密码验证，但 OAuth 注册用户（Google/Apple/Facebook）从未设置过密码，如何完成身份验证？
建议：明确两种路径——密码用户输入密码确认，OAuth 用户通过重新 OAuth 授权或邮箱验证码确认
P0开发2. Facebook 无邮箱用户的后续生命周期未覆盖
位置：PRD 第 2.6 节 Facebook 登录 + 第 3.4 节 忘记密码
调研报告将"Facebook 无邮箱注册容错"列为 MVP 必做项，PRD 提到 Facebook 可能不返回邮箱，但未说明：(1) 无邮箱用户如何找回密码？忘记密码流程依赖邮箱发验证码，这条路走不通；(2) 无邮箱用户如何接收交易通知？(3) 引导绑定邮箱的时机和流程是什么？
建议：新增"Facebook 无邮箱用户处理"章节，定义注册后引导绑定邮箱的触发时机（首次下单 / 7天内提醒）、绑定流程、未绑定时的功能限制
P0开发3. OAuth 回调错误处理未定义
位置：PRD 第 2.4/2.5/2.6 节 Google/Apple/Facebook 登录
三个 OAuth Provider 的正常流程写得清楚，但回调失败场景完全缺失：用户在 Provider 页面点"拒绝授权"、网络中断导致回调超时、state 参数校验失败、Provider 返回错误码。开发不知道这些场景该跳转哪里、显示什么提示。
建议：每个 OAuth 章节补充"异常场景"表格，至少覆盖：用户拒绝授权、回调超时、state 不匹配、Provider 返回错误四种情况

P1（应该修复，影响质量）

P1开发4. 跨 Provider 邮箱冲突处理逻辑不明确
位置：PRD 第 2.4-2.6 节 OAuth 登录
用户 A 用 Google（邮箱 a@gmail.com）注册后，又用邮箱 a@gmail.com + 密码注册，会怎样？反过来呢？PRD 提到"邮箱已注册"检查，但未说明跨注册方式的冲突解决策略——是自动关联、提示已存在、还是允许创建独立账号？
建议：明确邮箱唯一性策略：同一邮箱只能对应一个账号，后注册的方式自动关联到已有账号（需验证身份），或提示"该邮箱已通过 XX 方式注册，请使用 XX 登录"
P1开发5. 风控"跨州登录"的基准位置如何确定
位置：PRD 第 6.1 节 风控检测规则
跨州登录触发邮箱验证，但"用户的常驻州"如何确定？注册 IP？最近 N 次登录 IP 的众数？用户 profile 中的地址？这直接影响风控判断逻辑的实现。
建议：定义基准位置的确定规则，建议"最近 5 次登录 IP 的州级定位众数"或"注册时 IP 定位的州"
P1开发6. 密码修改页面对 OAuth 用户的可见性
位置：PRD 第 5.1 节 密码修改
密码修改要求输入"当前密码 + 新密码"，但纯 OAuth 注册用户从未设置过密码。这个页面对他们是隐藏、置灰、还是转为"设置密码"流程？
建议：区分两种场景——有密码用户走"修改密码"，无密码用户走"设置密码"（只需输入新密码 + 确认），或者对无密码用户隐藏该入口
P1开发7. 验证码规格未定义
位置：PRD 第 2.8 节 邮箱验证码
验证码的位数、字符集（纯数字？字母+数字？）、有效期均未说明。开发需要这些参数才能实现。
建议：明确定义，建议 6 位纯数字、有效期 10 分钟
P1开发8. 协议版本检测失败的降级策略
位置：PRD 第 5.4 节 协议版本检测
说"每次打开 App 或刷新页面时检查"，但如果 /api/auth/consent/check 接口超时或返回错误，是阻断用户使用还是静默跳过？这是个高频调用，降级策略必须明确。
建议：非阻断——接口失败时静默跳过，下次再检查。避免因协议检测接口故障导致全站不可用
P1测试9. 并发会话行为未定义
位置：PRD 第 3.6 节 登录态保持 + 第 5.2 节 Token 刷新
用户同时在 PC 和 H5 登录，修改密码后另一端的 session 是否立即失效？注销账号时其他端是否同步踢出？测试无法编写多端场景的用例。
建议：明确策略——修改密码后所有其他 session 的 refresh_token 立即失效；注销账号后所有 session 立即失效
P1测试10. 验证码新旧覆盖规则未定义
位置：PRD 第 2.8 节 邮箱验证码
用户请求验证码 A，未使用，又请求验证码 B。此时 A 是否还有效？如果两个都有效，存在安全风险；如果 A 失效，需要明确告知用户。
建议：新码生成时旧码立即失效，同一时刻只有一个有效验证码
P1测试11. 会话失效时的用户体验未说明
位置：PRD 第 5.3 节 会话失效
说"Token 过期后跳转登录页"，但如果用户正在填写表单（如注册到一半、修改密码到一半），跳转后表单数据丢失。是否需要保存草稿或给出提示？
建议：至少在跳转前 Toast 提示"登录已过期，请重新登录"，并在重新登录后尝试跳回原页面

P2（建议改进）

P2开发12. 国家选择器的异常场景
位置：PRD 第 1.4 节 国家/地区选择器
IP 检测失败时默认选哪个国家？检测到不支持的国家（如中国）时如何处理？支持的国家列表在哪里定义？
建议：IP 检测失败默认 United States；不支持的国家显示提示"Service not available in your region"
P2测试13. 浏览器后退按钮行为
位置：全局
多步骤流程（注册→验证码→成功、忘记密码→验证码→重置）中，用户点浏览器后退按钮的行为未定义。注册成功后后退到注册页，再次提交会怎样？
建议：关键成功页面使用 history.replaceState 替换历史记录，防止后退到已完成的步骤
P2开发14. 邮件重置密码链接 vs 验证码方式不一致
位置：PRD 第 3.4 节 忘记密码 vs 调研报告
调研报告 MVP 清单写的是"忘记密码（邮件重置链接）"，但 PRD 实现的是验证码方式。两边不一致，需确认最终方案。
建议：以 PRD 为准（验证码方式），同步更新调研报告描述

P3（小建议）

P3测试15. 接口清单缺少请求/响应示例
位置：PRD 第 7.2 节 接口清单
19 个接口只有路径和方法，没有请求参数和响应格式示例。虽然标注了"参考"，但测试编写接口测试用例时缺少依据。
建议：至少为核心接口（register、login、token/refresh）补充请求体和响应体示例
