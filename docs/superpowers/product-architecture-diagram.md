# 产品架构图绘制方法

## 绘制流程（必须按顺序执行）

1. **梳理业务域边界**：先用一句话定义域的边界，格式为：**【域名】负责【核心职责】，不负责【明确排除项】。** "负责"用动词短语说清做什么事，"不负责"列最容易被误塞进来的能力起防线作用，不写技术实现只写业务职责。写完发给相关人确认后再往下走。同时必须明确项目的业务场景特征（海外/国内、toB/toC、合规要求、多语言/多币种等），写成 checklist，后续所有归属判断都要对照这个 checklist 做，不能套用通用规则脱离项目背景
2. **架构抽象**（从功能点到系统机制，不能跳过）：
   - **理解业务本质**：不是问"要做哪些功能"，而是问：这个系统的核心价值是什么？数据从哪来、到哪去？不同业务之间的共性和差异是什么？
   - **从功能点归纳系统机制**：对每组功能点问三个问题：①这些功能点背后是同一个机制在驱动吗？②这个机制的输入是什么、输出是什么？③这个机制能独立变化吗？能独立变化 → 是独立的系统能力
   - **确定能力之间的层级关系**：不是平铺并列，而是问：谁依赖谁？谁支撑谁？信息流方向是什么？
   - **验证抽象质量**：①每个能力项能用一句话说清"输入→处理→输出"吗？不能 → 抽象不够清晰；②能力之间有没有重叠？有 → 合并；③层级关系能画出清晰的信息流吗？不能 → 层级划分有问题
   - **供给测试**：架构抽象完成后，必须检查：这个系统的数据/能力，有哪些其他系统在消费？这些消费方式需要系统主动提供能力支撑吗？需要 → 必须体现。适用于所有「基础数据型」系统（商品、用户、商家等），它们既有内部管理职责，又有对外供给职责，只看"系统内部做什么"会遗漏"系统对外提供什么"
   - **禁止从调研功能清单直接搬**：调研清单回答"要关注什么"，架构图回答"系统怎么组织这些能力"。每个功能点必须经过归纳和抽象，变成系统机制后才能上图
3. **列出候选模块清单**：基于架构抽象的结果列出模块，不是从调研清单直接搬
4. **逐一做归属判断**：每个模块过"拿走测试"，不属于的剔除或归入外围系统。只在模块级别对照业务场景 checklist
5. **确定分层结构**：根据实际内容决定需要哪些层，不要为了填满层级而凑内容
6. **确定核心能力的子层拆分方式**：根据当前域特点选择拆法，不默认套用"业务能力+技术能力"
7. **画图**：按确定的分层和模块开始绘制。先画骨架（层级+模块名），确认结构没问题后再填功能项细节
8. **画完回检**（强制）：
   - 正向：图里每个模块重新过一遍归属判断
   - 反向：回到域边界定义，看有没有核心能力被遗漏
   - 外围：每个外围系统项确认是真实运行时依赖
   - 供给链完整性：对每个功能项里涉及"引用/关联/触发"其他数据的地方，逐条追问——被引用的数据在图里有对应的管理模块吗？没有 → 补模块或确认由外部系统提供。典型案例：属性管理设计了"选品牌触发属性组"，但品牌主数据没有管理模块 → 遗漏

## 分层判断标准

不是所有层都要画，按以下标准判断是否需要体现：

1. **业务层**（最顶层） — 支撑多个业务模式时体现。如果模块耦合业务逻辑较少（如登录注册），不需要体现
   - **回答的问题**：系统服务什么业务？
   - **放什么**：业务类型/业务线（如二手奢侈品、二手消费电子）
   - **不放什么**：商家类型、运营模式、渠道类型。"三方商家""自营商家"是运营模式不是业务类型，属于商家系统范畴
   - 规划中的业务线可以体现，用虚线边框区分
2. **用户层** — 涉及多个用户类型或多个终端时体现
   - **回答的问题**：谁通过什么终端使用系统？做什么？
   - **放什么**：用户角色 + 该角色的关键操作（抽象到能力域层面，如"商品主数据管理"，不列具体功能如"编辑类目树"）
   - **布局方式**：按角色分组，角色名作为小标题，操作项横铺为等宽标签块（不用卡片形式）
   - **不放什么**：具体功能项。"商品发布""商品列表页"等是核心能力的入口，不是用户层的内容。功能项属于核心能力层
   - **区分「直接使用」和「消费数据」**：只放直接操作当前系统能力的用户角色。其他系统的用户虽然消费当前系统的数据（如买家看商详页），但操作主体不是当前系统，不放用户层。典型案例：买家端消费商品数据，但商详页/列表页/购物车属于各自前端模块，买家不直接使用商品系统
3. **解决方案**（可选） — 基于系统核心能力打包对外提供的组合方案时体现。模块本身就是核心能力则不需要
4. **核心能力** — 模块的核心能力，内部子层根据具体系统拆解，不固定
   - 常见拆法：业务能力 + 技术能力、核心能力 + 扩展能力、在线能力 + 离线能力
   - 选择依据：看当前域的能力天然分成哪两类，不要硬套
   - **子层最多3个**。超过3个时回到第一步审视域边界是否太宽。3个以内如果单个子层模块较多，可在子层内部用视觉分组（虚线框或间距）归类，避免平铺过散
   - 子层之间用虚线分隔
5. **外围系统** — 依赖的其他系统能力和三方外部系统
   - 内部系统：项目内已有或将建的其他域（如订单系统、营销系统）
   - 三方外部系统：需要对接的外部服务商
   - 边界不清时，问"这个系统的代码是我们团队维护的吗？"来区分内外
   - **准入标准**：只放当前域运行时必须调用的。判断方法：如果这个外部系统挂了，当前域的某个核心功能会不会不可用？会 → 放；不会 → 不放

关键原则：**有符合的内容才体现对应层级，没有就不画**

**层级准入检查**：每个独立层级必须代表一组真实的系统能力（有业务逻辑、有交互、有输入输出）。纯数据预留（只加字段不做交互）、纯规划（没有确定的能力项）、纯技术实现细节（数据库设计决策）不构成独立层级。如果某层的内容拿掉后不影响系统能力的完整性，说明它不够格独立成层

## 模块归属判断（强制检查项）

每个模块上图前必须通过以下检查：

1. **拿走测试**："这个能力从当前域拿走，当前域在实际业务场景下还能不能正常运营？"不是看技术上能不能跑通，而是看业务上能不能合法、合规、完整地运营。能正常运营 → 不属于当前域
2. **完整流程测试**：这个能力的完整执行链路，当前域自己能闭环吗？不能闭环 → 看当前域是"主控方"还是"配合方"。主控方 → 归属当前域；配合方 → 归属主控系统，当前域只保留配合动作（如接收指令执行下架）。典型案例：CPSC 召回拦截涉及商品下架+订单关闭+退款+通知，商品系统只是配合执行下架，不是主控方
3. **区分"关注"和"负责"**：调研阶段的功能清单回答"当前域需要关注什么"，架构图回答"当前域负责什么"。从调研到架构，每个功能项必须重新过职责边界判断，不能直接搬
4. **区分"属于"和"相关"**：有数据交互不等于属于。搜索消费商品数据但不属于商品域，推荐用到商品数据但不属于商品域
5. **前端展示层放用户层，不放核心能力**：商详页、列表页等用户触点属于用户层，体现用户通过什么入口使用系统能力，不是后端域的核心能力
6. **通用平台能力先分类再定位**：区分业务使用型（如搜索、推荐，业务流程中主动使用）和服务支持型（如消息推送、日志、监控，底层支撑）。业务使用型放业务层，服务支持型放外围系统（最底层）
7. **技术实现细节不上图**：Elasticsearch、Redis、具体加密算法等属于技术选型
8. **通用基础不上图**：日志审计、数据加密、缓存等

## 功能项上图前逐条检查（每个子项必过）

每写一个功能项之前，必须逐条过以下检查，全过才上图：
1. **拿走测试**：这个能力从当前域拿走，业务还能正常运营吗？能 → 不属于当前域
2. **完整流程测试**：这个能力的完整执行链路，当前域自己能闭环吗？能闭环 → 归属当前域；不能闭环 → 看当前域是"主控方"还是"配合方"。主控方 → 归属当前域；配合方 → 不上图，当前域只保留配合动作（已包含在状态流转等已有能力中）
3. **来源审查**：这个功能项是从调研清单直接搬过来的，还是经过架构抽象重新定义的？直接搬的 → 必须重新过职责边界判断，区分"关注"和"负责"
4. **通用能力判断**：这是当前域独有的，还是多个域都需要的？多个域都需要 → 放外围系统或不放
5. **禁止占位**：具体场景是什么，当前阶段需要吗？说不清 → 不上图，记入"待确认清单"。确认后再补上图，避免遗漏
6. **层级检查**：和模块名、其他子项有重复或包含关系吗？有 → 合并或去掉
7. **业务确认**：涉及业务逻辑的，实际业务设计确认了吗？没确认 → 先确认再放
8. **命名检查**：别人看到这个名字会有歧义吗？会 → 加限定词或括号说明
9. **合规/法规需求拆解**：如果功能项来源于合规或法规要求，不能直接映射成模块。必须先问：①这个要求在系统里对应什么具体动作？②这个动作是哪个系统的职责？③是否已被现有模块覆盖？典型案例："二手/翻新状态强制标注"拆解后 → 成色字段必填校验（已被属性模板覆盖）+ 翻新商身份披露（商家系统职责），不需要单独建模块

## 功能项命名规范（消除歧义）

- **每个能力项必须自解释**：别人看到这个词，不能产生两种以上的理解
- **有适用范围的必须标注**：用括号说明适用对象/场景，如 `账号密码登录（邮箱/手机号）`
- **有关联关系的能力项，命名要体现边界**：不能让读者自己推理哪些能力之间有依赖或互斥关系
- **禁止笼统概念占位**：不能用模糊的大词（如"免密登录"）代替具体能力（如"Passkey 登录"）
- **涉及业务逻辑的功能项必须先确认**：不能套用行业通用模式，必须基于项目实际业务设计来判断
- **功能项不能有包含或重叠关系**：同一卡片内的功能项之间如果存在包含、从属或重叠关系，必须合并。功能项也不能和模块名含义重复
- **画完回检时增加歧义检查**：逐个能力项问"别人看到这个词，会不会理解成别的意思？"

## 功能项粒度标准

- 优先统一到"用户可感知的独立操作"这一层
- 无法按用户感知判断时（如后端/技术支撑类能力），用以下两条检验：
  1. 如果某个功能项还能拆出2个以上独立子项，而同图其他功能项拆不出来 → 粒度太粗，需要拆
  2. 如果某个功能项是另一个功能项的步骤或子动作 → 粒度太细，需要合并
- 同一张图内所有卡片的功能项粒度必须对齐

## 跨域一致性

- 维护一份"公共依赖清单"，记录哪些外围系统被多个域依赖
- 新画一个域的架构图时，对照清单检查是否遗漏共同依赖

## 颜色体系

- **画布底色**：浅灰 #F9FAFB（所有架构图统一）
- **标题栏**：深藏蓝 #1E1F2E（所有架构图统一，不用渐变）
- **层级底色**：全部白底 #fff + 浅灰边框 #E5E7EB，不填充彩色底色
- **层级区分**：靠左侧彩色竖条 + 层标题颜色区分，不靠底色
- **卡片颜色按子层统一，不按功能域分**：同一子层内所有卡片用同一色系，靠标题文字区分模块
- 已定义的层级竖条颜色：
  - 业务层：紫色 #8B5CF6
  - 用户层：绿色 #10B981
  - 核心能力：蓝色 #4A6CF7
  - 外围系统：橙色 #F59E0B
- 已定义的卡片色系：
  - 蓝色系（边框 #60A5FA，标题栏 #DBEAFE，功能项 #EFF6FF/#93C5FD，文字 #2563EB/#1D4ED8）
  - 外围系统标签：暖黄 #FFFBEB/#FEF3C7，边框 #F59E0B，文字 #B45309/#92400E
- **层级超过4层时**：优先复用已有颜色，新增层从灰色系 #6B7280 扩展，避免颜色过多

## 布局与卡片规范

- **核心能力层统一用嵌套方块展示**：功能组框（浅色底+边框+组标题）内放白底标签，体现多层嵌套层级感。不用扁平标签或卡片形式
- **dual 子框内功能组框横排排列**：从左到右等宽并排，每个框内标签纵向排列，高度按最多标签数对齐
- **标准数据层功能组框横排排列**：同 dual 子框内逻辑，多列平铺
- **视觉层级通过底色交替实现**：白底（层框/子框）→ 浅色底（功能组框）→ 白底（标签），清晰区分嵌套层级
- **卡片结构匹配内容**：有子功能项 → 标题栏+功能项列表卡片；无子项 → 简洁单行标签，不做空壳卡片
- **卡片数量变化时**：等分宽度重新排列，保持视觉均衡
- **禁止布局空白**：外框高度紧贴内容，间距不超过 12px。增删内容后必须同步更新外框高度、viewBox、背景矩形、底部说明位置
- **配色标准统一**：同类元素必须用同一套标准，不能混用

## 输出格式

- 使用 SVG 直接绘制，浏览器可直接打开
- **必须嵌入双层结构化数据**：SVG 文件头部（`<svg>` 标签之后、`<defs>` 之前）必须包含两个数据块：
  1. **产品定义**（XML 注释 `<!-- STRUCTURED DATA ... -->`）：模块名、版本、范围、所有层级和功能项，用于画图渲染
  2. **技术编码规格**（`<metadata><arch-tech-spec format="yaml"><![CDATA[...]]></arch-tech-spec></metadata>`）：与产品定义一一对应，每个能力项额外包含技术字段（service/api/priority/tech_notes），产品侧预填 name，技术字段留空给开发补。外围系统额外包含 provider/interface/tech_notes 字段
  - 新建或更新产品架构图时，两个数据块必须同步生成/更新，不能只有产品定义没有技术规格
  - 技术规格的层级结构（layers → groups → items）必须与产品定义完全对应，不能有遗漏或多余
- 字体：PingFang SC / Microsoft YaHei / Helvetica Neue / Arial
- 层标题用 `font-weight="700" letter-spacing="2"`

## SVG 注意事项

- XML注释内禁止连续双连字符 `--`
- 只用 viewBox，不设固定 width/height
- 使用 filter + feDropShadow 做卡片阴影
- 标题栏用纯色 #1E1F2E，不用渐变
- 内容较多时分段写入（先上半部分，再追加下半部分），避免单次写入过大
- **改内容后必检**：同一行的卡片 height 值必须一致，逐行核对，不一致的统一到该行最大值。同一行卡片内的功能项排列方式必须统一（全部单行满宽或全部双列），不能混排
- **SVG 提交前强制核对**：写完 SVG 后，提交前必须逐项核对以下内容，全部通过才能提交：
  1. 同行卡片的 height 值是否一致
  2. 同行卡片内功能项排列方式是否统一
  3. 外围系统所有子项的 width、height 是否统一
  4. 同层级元素的尺寸（width/height）是否一致
- **外围系统布局规范**：依赖的平台能力和三方外部系统必须左右布局，不能上下排列。两组子标签和子项的 y 坐标、height 值必须一致。所有子项方块的 width 必须统一，不能有长有短
