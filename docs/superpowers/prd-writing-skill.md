# PRD 写作方法

## 输出格式

- 默认输出 Markdown（.md），文件命名：`looply-{模块名}-PRD-v{版本号}.md`
- 如用户要求生成 Word，再用 python-docx 生成 .docx（含自动目录 TOC）
- 标题层级：最多三级
- 存放位置：`~/Desktop/海外业务{模块名}/PRD/`

## 文件存放规则

PRD 相关的文件按以下路径组织：

- PRD 文档：`~/Desktop/海外业务{模块名}/PRD/`
- 调研文档：`~/Desktop/海外业务{模块名}/调研/`
- 评审记录：`~/Desktop/海外业务{模块名}/评审记录/`
- 验收记录：`~/Desktop/海外业务{模块名}/验收记录/`
- 交付开发：`~/Desktop/海外业务{模块名}/交付开发/`（存放给开发的最新文件，每次产出更新时同步复制到此文件夹）
- 交付包命名：`交付开发 V{版本号}-{日期}`（如 `交付开发 V1.0-20260420`），版本号与变更日志对齐
- 新生成的文件直接放对应文件夹，不要放桌面根目录

## 生成流程（必须按顺序执行）

### 第一步：输入源收集与交叉对齐（硬性检查点）

动手写之前，必须先完成以下检查：

1. **盘点输入源**：列出当前模块已有的所有输入源
   - 产品架构图（功能范围依据）
   - 系统流程图（流程逻辑依据）
   - 实体关系图（数据模型依据）
   - UI 设计稿（界面依据，PC 端 + APP 端）
   - 模块启动包调研产出（`调研/` 目录下的竞品对标、合规扫描、服务商选型、需求优先级、模块启动报告）
   - 竞品分析、用户调研等（如有）

2. **交叉比对**：将所有输入源逐项对照，输出覆盖差异表
   - 格式示例：「流程图 9 段 vs 设计稿 15 页，缺密码修改和协议检测」
   - 每个差异标注：遗漏 / 当前不做 / 待确认

3. **用户确认**：将差异表发给用户，确认每个缺口的处理方式后再动手
   - 不能默默标"待设计"跳过

4. **调研产出引用**：如果模块有 `调研/` 目录，自动读取以下文件作为 PRD 写作上下文：
   - `模块启动报告.md`（全局参考）
   - `合规扫描.md`（写"兼容性与约束"章节时参考）
   - `需求优先级.md`（写"版本规划"章节时参考）
   - 不强制要求有调研产出，没有则跳过

### 第二步：确定 PRD 结构

文档按业务模块组织，每个模块自包含功能描述、交互、校验、异常、UI 关联。功能说明是核心，占全文 60% 以上。

每个章节标注「必写」或「按需」。按需章节标注了可省略的条件，满足条件时可跳过，但需在文档中注明"本模块不涉及，故省略"。

修订记录不放正文，单独维护在 `looply-{模块名}-PRD-修订记录.md` 中，与 PRD 正文同目录。格式固定为：`| 版本 | 日期 | 修改人 | 修改内容 |`

```
一、概述【必写】
   1.1 背景与目标
   1.2 不做什么（明确边界）
   1.3 用户角色
   1.4 核心场景
   1.5 全局页面流转
   1.6 术语说明
   1.7 多语言 / 多国家策略

二、需求详细描述【必写】
   公共功能组件和各业务模块平级排列。
   公共功能组件按需放在前面（写完业务模块后，在自检环节检查是否有重复组件需要提取）。
   每个模块/组件包含「模块概述」+ 若干功能项。
   功能项根据类型选用不同子结构（见下方三套子结构模板）。

三、依赖与风险【必写】
   - 上下游系统依赖
   - 外部服务依赖（三方 OAuth、邮件服务商等）

四、版本规划【必写】
   - 当前版本（MVP）范围
   - 后续迭代方向

五、数据与埋点【按需：有埋点需求时写】
   - 关键埋点事件
   - 数据字段与来源

六、权限与角色矩阵【按需：涉及多角色差异化权限时写】

七、附录【必写】
   - 设计稿索引（全量页面 × PC/APP 设计稿对照表）

注意：附录中不写接口清单，API 接口属于技术文档范畴，不属于 PRD，减少对开发系统设计的干扰
```

#### 全局页面流转说明

全局页面流转只展示**页面之间的跳转关系**（从哪个页面到哪个页面），不包含业务逻辑判断。目的是让读者快速了解"一共有哪些页面、怎么跳"。用页面名称 + 箭头即可，不需要画完整流程图。

与系统流程图的区别：
- 全局页面流转 = 页面导航地图（只看页面）
- 系统流程图 = 完整业务逻辑（含判断、校验、后端处理）

#### 功能项子结构模板（按类型选用）

写每个功能项时，根据功能类型选用对应的子结构。不要求所有功能统一格式。

**类型判断规则**：有独立页面 → 页面型；无独立页面、串联多个页面 → 流程型；无页面 → 机制型。如果一个功能既有独立页面又涉及跨页面跳转（如注册），用页面型，在操作流程中写清跳转关系。

**页面型**（有独立页面的功能，如注册页、登录页）：
```
功能描述
前置条件
页面布局（PC 端 / APP 端差异）
页面元素
页面状态变体（默认态/错误态/加载态/置灰态等，列表说明 + 对应设计稿）
操作流程（主流程 + 分支流程）
校验规则（字段 × 规则 × 校验时机）
交互说明
异常处理
UI 关联（PC 端设计稿 + APP 端设计稿）
```

**流程型**（跨页面的业务流程，如忘记密码、OAuth 认证）：
```
功能描述
前置条件
操作流程（主流程 + 分支流程，含页面跳转说明）
校验规则
异常处理
技术说明（按需，如 OAuth 授权码流程等需要开发理解的机制）
UI 关联
```

**机制型**（无独立页面的后台机制，如 Token 刷新、会话失效、登录限错）：
```
功能描述
触发条件
处理流程（含各场景的处理方式）
规则说明（如限频规则、有效期规则等）
异常处理
技术说明
```

### 第三步：撰写内容

写作原则：

1. **行为描述代替技术术语**：用"自动延长有效期"而非"滑动续期"，用"系统记住登录状态"而非"写入 Session"，让产品、开发、测试、运营都能理解
2. **每个功能模块必须有 UI 关联**：PC 端和 APP 端都要列出对应设计稿页面名称
3. **规则要具体可执行**：不写"密码需要一定强度"，写"密码 8-64 位，至少包含大写字母、小写字母、数字中的两种"
4. **异常流程不能省**：每个功能至少覆盖：网络异常、输入非法、权限不足、数据不存在、并发冲突
5. **海外业务特殊关注**：
   - 多国合规（GDPR 数据处理、CCPA 隐私权利、Cookie 同意）
   - 多语言文案（是否需要 i18n key）
   - 多币种 / 多时区处理
   - 二手商品特有逻辑（质检信息、成色等级、渠道差异定价）

### 第四步：自检（写完后强制执行）

1. **章节完整性检查**：概述、需求详细描述、兼容性与约束、依赖与风险、版本规划、附录是否都有。标注「必写」的章节不能缺失，「按需」的章节如省略需注明原因
2. **路径完整性检查**：按用户操作路径从头走一遍，确保每个页面、每个状态都有对应章节
3. **目录与正文一致性**：目录索引和正文标题一一对应，无遗漏无多余
4. **章节归属合理性**：检查每个小节是否放在了正确的父章节下。跨模块复用的功能组件放在「需求详细描述」章节内的公共功能组件部分，不放某个业务模块下
5. **重复组件检查**：检查各业务模块中是否有重复描述的功能组件（如同一个验证码页面在多处出现），如有则提取到公共功能组件中，各业务模块改为引用
6. **功能项类型匹配检查**：每个功能项是否选用了合适的子结构模板（页面型/流程型/机制型），子结构中的必要段落是否齐全
7. **UI 关联完整性**：逐个功能模块检查，PC 端和 APP 端的设计稿引用是否都有。每次修改 PRD 后必须全量检查一遍 UI 关联是否完整
8. **待设计页面检查**：发现有页面/状态尚未设计（无设计稿），必须停下来提醒用户确认处理方式（补设计、延后、砍掉等），不能自行标"待设计"继续写
9. **外部服务对接检查**：检查流程中是否涉及外部服务对接（如邮件服务、三方 OAuth、支付、短信等），如有则提醒用户确认：服务商选型、开发者账号注册、合规材料准备（隐私政策/服务条款）、审核周期等前置依赖是否已安排
10. **设计稿交叉校验**：与设计稿逐页对照文案，确保两边一致
11. **输入源回检**：回到第一步的覆盖差异表，确认所有"需覆盖"的项都已写入 PRD
12. **术语一致性**：全文同一概念用同一个词，不要一会儿叫"商品"一会儿叫"产品"

## 开发反馈处理流程

收到开发反馈时，分两步处理：

1. **确认改动方案**：先调用 `dev-feedback` skill 分析反馈内容、评估影响范围、与用户确认改动方案
2. **修改 PRD**：方案确认后，回到本 skill 按第三步（撰写内容）和第四步（自检）的规则修改 PRD 对应章节

## 注意事项

- PRD 是给人看的文档，不是技术规格书。优先保证可读性
- **禁止写接口清单**：API 接口（路径、方法、参数）属于技术文档范畴，PRD 只描述功能和业务规则。包括附录也不写
- 版本记录要写清楚每次改了什么、为什么改，方便追溯
- 功能说明是核心，篇幅应占全文 60% 以上
- 不确定的需求先跟用户确认，不要自己假设后写进 PRD

## 需求版本管理规范

PRD 交付后需配合版本管理，规则如下：

- **适用范围**：所有模块（商品、登录注册、订单、支付等），新模块首次交付时按此规范建立变更日志
- **V1.0 前例外**：在交付开发 1.0 版本前，暂不需要建立或维护变更日志；从首次交付开发 V1.0 之后，再按标准版本管理规范执行
- **版本管理 skill**：优先使用 `req-versioning` skill，旧入口 `version-management` / `delivery-diff-summary` 视作历史来源
- **模式划分**：
  - 模式 A（迭代留痕）：修改产品架构图、实体关系图、系统流程图、原型、PRD 后持续更新变更日志
  - 模式 B（交付收口）：准备交付开发时基于前后实际交付文件生成核心差异汇总
- **记录口径**：只要 PRD、原型、架构图、ER 图、流程图等实际产出发生改动，都应进入变更日志；交付收口时基于前后实际交付文件覆盖所有实际改动范围
- **交付包命名**：`交付开发 V{版本号}-{日期}`（如 `交付开发 V1.1-20260425`），版本号与变更日志对齐
