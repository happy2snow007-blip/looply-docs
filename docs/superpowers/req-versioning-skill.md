# 需求迭代版本管理

统一承接两类工作：
- **模式 A：迭代留痕** —— 修改 PRD、原型、架构图、ER 图、流程图后，持续记录到当前进行中版本
- **模式 B：交付收口** —— 准备交付开发时，基于前后交付文件 + 当前版本日志，生成最终差异汇总和开发关注重点

这是 `version-management` 与 `delivery-diff-summary` 两个旧入口合并后的主入口。后续优先使用本 skill。

## 适用范围

- 适用于所有模块（商品、登录注册、订单、支付等），新模块首次交付时按此规范建立变更日志
- **V1.0 前例外**：在交付开发 V1.0 版本前，暂时不需要建立或维护变更日志；从首次交付开发 V1.0 之后，再按标准版本管理规范执行

## 触发时机

### 模式 A：持续记录
当修改以下任何产品产出文件时使用：
- 产品架构图（.svg）
- 实体关系图（.svg / .html）
- 系统流程图（.svg）
- 原型（.html / .pen）
- PRD（.md / .docx）

### 模式 B：交付收口
当准备给开发交付新版本，需要输出：
- 本次版本相对上一交付版本的核心差异
- 开发需重点关注的范围
- 可直接放入交付包的差异汇总文档

## 输出位置

- 变更日志：`~/Desktop/海外业务{模块名}/looply-{模块名}-变更日志.md`
- 交付包：`~/Desktop/海外业务{模块名}/交付开发 V{版本号}-{日期}/`
- 差异汇总：`~/Desktop/海外业务{模块名}/交付开发 V{版本号}-{日期}/V{上一版本}-V{本次版本}-核心差异汇总.md`

## 文件发现机制

通过目录约定自动定位：
- 模块根目录：`~/Desktop/海外业务{模块名}/`
- 常见子目录：`PRD/`、`UI/`、`产品架构图/`、`实体关系图/`、`系统流程图/`、`交付开发/`
- 变更日志命名：`looply-{模块名}-变更日志.md`

## 执行流程

### Step 1：识别当前工作模式

先判断当前任务属于哪一类：

**进入模式 A 的信号**：
- 用户刚修改了 PRD / 原型 / 图稿
- 用户说“更新变更日志”“记录这次迭代”“同步版本记录”

**进入模式 B 的信号**：
- 用户说“准备交付开发”“生成本次差异说明”“做交付汇总”
- 当前目录下已经出现新的交付包版本

如果不明确，先问用户：
- 这次是要**持续记录进行中版本**，还是要**生成交付版差异汇总**？

## 模式 A：持续记录进行中版本

### Step A1：检测变更日志是否存在

- 检查模块目录下是否已有 `looply-{模块名}-变更日志.md`

**如果不存在**：
- 参考模板创建新的变更日志
- 参考下方「变更日志模板」章节创建
- 如尚未有 V1.0 基线，遵守 memory 规则：**V1.0 前可暂不建日志；V1.0 后按规范执行**

**如果存在**：
- 读取当前变更日志
- 识别当前“进行中”版本号（如 V1.2）
- 识别上一已交付版本号（如 V1.1）

### Step A2：识别文档类型和变更内容

根据文件名和路径识别：
- 文档类型：原型 / PRD / 产品架构图 / 实体关系图 / 系统流程图
- 所属模块：商品 / 登录注册 / 订单等
- 文件版本变化：如 `v14 → v15`、`V2.5 → V2.6`
- 本次变化属于：新增交付物 / 已有交付物调整

### Step A3：判断变更类型

**新增交付物**：
- 该文档类型在上一已交付版本中标记为“未交付”
- 或该文档类型第一次进入本模块交付范围

**变更内容**：
- 该文档类型在上一已交付版本中已经交付
- 本次是对既有交付物的增量修改

### Step A4：更新进行中版本日志

#### A4.1 新增交付物

在“### 新增交付物”章节下记录：

```markdown
#### {文档类型}
- **文件名**：{完整文件名}
- **状态**：🔄 迭代中（如用户确认已定稿则改为 ✅ 稳定）
- **主要内容**：
  - {本次新增内容}
- **说明**：首次纳入交付范围
```

#### A4.2 已有内容调整

在“### 变更内容”章节下：

**如果是新功能模块**：

```markdown
##### 🔄 {功能名}（迭代中）
- **{文档类型}版本**：v{旧版本} → v{新版本}
- **状态说明**：功能还在迭代调整中，暂不交付开发
- **新增内容**：
  - {本次新增/调整点}
- **功能说明**：
  - {用途与场景}
- **迭代记录**：
  - {日期}：{本次变更说明}
```

**如果是已有功能调整**：
- 在对应功能项的“迭代记录”下追加本次变更
- 不要重写整段历史，只记本次增量

### Step A5：更新影响范围与待办事项

在“### 影响范围”中更新：

```markdown
- **{文档类型}**：已更新（v{新版本}）
- **其他文档**：待同步更新 / 无影响
```

在“### 待办事项”中追加：

```markdown
- [ ] {需要补齐或同步的工作}
```

## 模式 B：交付收口生成差异汇总

### Step B1：识别交付基线

必须先确认：
- 上一已交付版本号（如 V1.0）
- 本次准备交付版本号（如 V1.1）
- 两个交付包目录位置

**核心原则**：
- 基于**实际交付文件**对比，不基于记忆、中间迭代或口头描述
- 只有“上一版有、本次没有”的内容，才算删除
- 只看最终交付边界，不把中间试错版本当成正式输入

### Step B2：读取前后交付范围

读取并整理：
- 上一交付包内的 PRD / 原型 / 图稿 / 其他正式交付物
- 本次交付包内的 PRD / 原型 / 图稿 / 其他正式交付物
- 当前版本日志中的全部实际变更记录（包含 ✅ / 🔄 / 📝），用于辅助理解迭代过程与影响范围

### Step B3：生成核心差异结构

按以下维度整理：
- 删除功能
- 新增功能
- 既有功能调整
- PRD 章节变化
- 页面 / 原型变化
- 图稿变化（架构图 / ER 图 / 流程图）
- 开发关注重点
- FAQ / 风险提醒（如有）

输出格式建议：

```markdown
# V{上一版本} → V{本次版本} 核心差异汇总

## 1. 本次交付范围
- PRD：...
- 原型：...
- 产品架构图：...
- 实体关系图：...
- 系统流程图：...

## 2. 删除功能
- ...

## 3. 新增功能
- ...

## 4. 既有功能调整
- ...

## 5. 开发关注重点
- ...

## 6. 需同步关注的依赖项
- ...
```

### Step B4：保存到交付包目录

将差异汇总保存到本次交付包文件夹，命名：
- `V{上一版本}-V{本次版本}-核心差异汇总.md`

并确保交付包命名符合规范：
- `交付开发 V{版本号}-{日期}`

## 功能成熟度标记

每个功能模块可以标注成熟度，用于表达当前确定性：
- ✅ **稳定**：当前方案相对明确
- 🔄 **迭代中**：还在持续调整
- 📝 **草稿**：刚开始设计，变化可能较大

默认规则：
- 新增功能默认 `🔄 迭代中`
- 只有用户明确确认定稿，才改为 `✅ 稳定`
- 状态标记仅用于表达成熟度，不作为是否记录进变更日志、是否纳入最终差异对比的过滤条件
- 只要 PRD / 原型 / 图稿等实际产出发生改动，都应记录到变更日志，并在交付收口时纳入前后实际交付文件的差异分析范围

## 版本号规则

- **小版本递增**：`V1.0 → V1.1 → V1.2`，用于功能调整、页面优化、字段变更
- **大版本递增**：`V1.0 → V2.0`，用于架构重构、大模块新增、边界显著变化
- **交付包命名**：`交付开发 V{版本号}-{日期}`（如 `交付开发 V1.1-20260425`），版本号与变更日志对齐。V1.0 保持原名格式 `交付开发 {日期}`（如 `交付开发 20260420`）不改，从 V1.1 开始统一带版本号

````markdown
## 使用说明

### 版本号规则
- 小版本号递增（V1.0 → V1.1 → V1.2）：功能调整、页面优化、字段变更
- 大版本号递增（V1.0 → V2.0）：架构重构、大功能模块新增

### 功能成熟度标记
每个功能模块用标记区分当前状态：
- ✅ **稳定**：已定稿，可以开始开发
- 🔄 **迭代中**：还在调整，暂时不要动
- 📝 **草稿**：刚开始设计，变化很大

### 变更记录格式

#### 迭代过程中
- 每个功能模块独立记录，标注成熟度状态
- 迭代中的功能：在"迭代记录"下追加每次调整（记增量）
- 稳定的功能：记录最终方案（全量描述）

#### 功能定稿时
- 把迭代记录合并成"最终方案"（开发只需看最终结果）
- 迭代历史折叠保留，标注"开发可跳过"
- 状态从 🔄 改为 ✅，更新"状态说明"

#### 冲突处理
当迭代中的功能影响了已稳定的功能时：
1. 方案 A：把稳定功能降级回 🔄，标注原因
2. 方案 B：在稳定功能下追加"待调整"记录，标注依赖关系

### 交付流程
1. 在"进行中"版本下持续记录变更
2. 只有标记为 ✅ 稳定的功能才能交付开发
3. 攒到合理批次后，标记"已交付"并更新交付日期
4. 将最新文件复制到交付开发文件夹
5. 交付包命名规范：`交付开发 V{版本号}-{日期}`
6. 通知开发团队查看变更日志

### 开发视角
开发只需关注：
- 从上次交付版本到当前版本的所有"已交付"条目
- 只看标记为 ✅ 稳定的功能（🔄 迭代中的功能暂时不动）
- 每条变更的"影响范围"（哪些实体/接口/页面受影响）
- 是否需要调整已写的代码
````

进行中版本至少包含以下章节：
- `### 新增交付物`
- `### 变更内容`
- `### 影响范围`
- `### 待办事项`

## 执行守则（防遗漏清单）

每次使用 `req-versioning`，至少逐项确认以下内容：

### 通用
- [ ] 先判当前是模式 A 还是模式 B
- [ ] 已确认模块名、当前版本号、上一已交付版本号、交付边界
- [ ] 所有判断均基于实际文件，不靠猜测、记忆或口头描述
- [ ] 成熟度标记仅作成熟度说明，没有被当成记录范围或差异范围的过滤条件

### 模式 A：迭代留痕
- [ ] 已确认当前进行中版本和上一已交付版本
- [ ] 已识别文档类型、所属模块、文件版本变化
- [ ] 已判断本次属于“新增交付物”还是“已有内容调整”
- [ ] 变更日志记录的是实际增量内容，而不只是文件名或“已更新”字样
- [ ] 已更新影响范围
- [ ] 已更新待办事项

### 模式 B：交付收口
- [ ] 仅基于前后实际交付文件做对比
- [ ] 删除功能判断满足“上版有、本版无”
- [ ] 新增功能不只写交互细节，也覆盖业务范围、章节范围、页面范围、图稿范围
- [ ] 已覆盖 PRD / 原型 / 图稿 / 依赖项的变化
- [ ] 最终输出对开发有明确行动指引，能看出哪些要新做、哪些要改、哪些要同步关注

## 自检清单

1. 已识别当前是模式 A 还是模式 B
2. 已确认模块名、版本号、交付边界
3. 变更日志不存在时已按模板创建，存在时已正确识别进行中版本
4. 已判断是“新增交付物”还是“已有内容调整”
5. 所有变更记录都基于实际文件，不靠猜测
6. 成熟度标记仅用于表达当前确定性，未被误用为记录或对比范围的过滤条件
7. 影响范围与待办事项已同步更新
8. 交付收口时，差异汇总基于前后交付包实际文件对比，并覆盖所有实际改动范围
9. 输出文件已保存到正确目录
10. 版本号与交付包命名一致

## 反模式

1. **只记最终文件名，不记变更内容**：开发无法理解本次到底改了什么
2. **拿中间迭代版本当交付基线**：会导致删除/新增判断失真
3. **把成熟度标记误当成交付记录或差异范围过滤条件**：会错误遗漏实际已变更的产出
4. **凭记忆写差异，不核对实际文件**：容易遗漏页面、章节或图稿
5. **日志和交付包版本号不一致**：后续无法追溯
6. **继续推荐旧 skill 入口**：合并后应优先使用 `req-versioning`
