# 登录注册模块合规扫描

## 扫描范围

登录注册模块涉及的业务行为：
- 用户注册（收集邮箱、密码、国家/地区）
- 三方 OAuth 登录（Google/Apple/Facebook 数据获取）
- 邮箱验证（发送验证邮件）
- 密码存储与重置
- 登录状态管理（Token/Session）
- 账号注销与数据删除
- 营销邮件订阅（注册时 opt-in）

## 1. 美国联邦层面

### 1.1 FTC Act Section 5 — 数据安全

| 项目 | 内容 |
|------|------|
| 法规 | FTC Act Section 5（15 U.S.C. § 45） |
| 级别 | **必须遵守** |
| 核心要求 | 禁止"不公平或欺骗性"商业行为，要求对消费者数据实施"合理"安全措施 |
| 对登录注册的影响 | |

**必须做**：
- 密码不能明文存储，必须使用行业标准哈希算法（bcrypt/Argon2id + 盐值）
- 传输层加密（HTTPS/TLS）
- 防止常见攻击（SQL 注入、XSS、CSRF）
- 隐私政策中承诺的安全措施必须实际执行（否则构成"欺骗性行为"）
- 数据最小化：只收集注册必需的信息

**参考**：FTC "Start with Security" 指南（ftc.gov/business-guidance/resources/start-security-guide-business）

### 1.2 COPPA — 儿童隐私保护

| 项目 | 内容 |
|------|------|
| 法规 | Children's Online Privacy Protection Act（COPPA） |
| 级别 | **必须遵守** |
| 核心要求 | 收集 13 岁以下儿童个人信息前，必须获得可验证的家长同意 |
| 对登录注册的影响 | |

**必须做**：
- 注册流程中加入年龄筛查（age gate）：询问出生日期或年龄确认
- 如用户表明未满 13 岁：阻止注册或获取可验证的家长同意
- 隐私政策中明确说明儿童数据处理方式

**建议做**：
- 参考 Vinted 做法：注册时声明"我已年满 18 岁"（looply 涉及交易，18+ 更合适）
- 2026年2月 FTC 新政策鼓励使用年龄验证技术，可关注后续要求

**注意**：looply 是二手电商平台，非儿童定向网站，但如果"实际知道"有 13 岁以下用户注册，COPPA 仍然适用。

### 1.3 CAN-SPAM Act — 商业邮件

| 项目 | 内容 |
|------|------|
| 法规 | Controlling the Assault of Non-Solicited Pornography And Marketing Act |
| 级别 | **必须遵守** |
| 核心要求 | 商业邮件必须提供退订机制，10 个工作日内处理退订请求 |
| 对登录注册的影响 | |

**必须做**：
- 注册时如果有营销邮件订阅选项，不能默认勾选（opt-in 而非 opt-out）
- 验证邮件属于"事务性邮件"（transactional），不受 CAN-SPAM 营销规则约束，但仍需准确标识发件人
- 所有营销邮件必须包含：物理邮寄地址、退订链接
- 退订请求必须在 10 个工作日内生效

**建议做**：
- 注册流程中营销邮件订阅与账号创建分离，不捆绑
- 验证邮件中不夹带营销内容

### 1.4 NIST SP 800-63B — 数字身份指南

| 项目 | 内容 |
|------|------|
| 标准 | NIST Special Publication 800-63B（Digital Identity Guidelines） |
| 级别 | **建议遵守**（非强制法规，但被广泛引用为行业标准） |
| 对登录注册的影响 | |

**密码策略建议**：
- 最低 8 字符，建议支持最长 64 字符
- 不强制复杂度规则（如"必须包含大写+数字+特殊字符"）——NIST 已明确反对此做法
- 检查密码是否在已泄露密码库中（如 Have I Been Pwned）
- 不强制定期更换密码
- 密码存储：使用 Argon2id（首选）或 bcrypt（work factor ≥ 10）+ 唯一盐值

**会话管理建议**：
- Session token 足够长且随机
- 支持并发会话管理（用户可查看/终止活跃会话）
- 敏感操作（改密码、注销）需重新验证身份

## 2. 州级法规

### 2.1 CCPA/CPRA — 加州消费者隐私法

| 项目 | 内容 |
|------|------|
| 法规 | California Consumer Privacy Act + California Privacy Rights Act |
| 级别 | **必须遵守**（适用条件：年收入 > $25M，或处理 10万+ 加州居民数据，或 50%+ 收入来自出售个人信息） |
| 生效 | CCPA 2020，CPRA 修正案 2023，最新修订 2026-01-01 生效 |
| 对登录注册的影响 | |

**必须做**：
- 注册前/注册时提供隐私政策链接，说明收集哪些数据、用途、是否出售/共享
- 提供"不要出售或共享我的个人信息"选项（Do Not Sell or Share）
- 用户有权：访问、删除、更正个人信息；知道数据被收集和使用的方式
- 账号注销时必须删除用户数据（或提供数据删除请求通道）
- 不能因用户行使隐私权而歧视（如拒绝服务或降低服务质量）
- 数据泄露通知义务

**注册流程具体要求**：
- 注册页面底部或显著位置放置隐私政策链接
- 如果注册时收集的数据会用于个性化推荐/广告，需在隐私政策中明确说明
- 提供数据删除请求的入口（可在账号设置页）

### 2.2 其他州隐私法（已生效或即将生效）

截至 2026 年，美国已有 20+ 个州通过了综合隐私法。与登录注册最相关的共性要求：

| 州 | 法规 | 生效日期 | 与 CCPA 的主要差异 |
|----|------|---------|-------------------|
| 弗吉尼亚 | VCDPA | 2023-01 | 无私人诉讼权，仅 AG 执法 |
| 科罗拉多 | CPA | 2023-07 | 要求通用退出机制（universal opt-out） |
| 康涅狄格 | CTDPA | 2023-07 | 类似 CCPA，含数据最小化要求 |
| 犹他 | UCPA | 2023-12 | 门槛更高（$25M 收入 + 10万用户） |
| 德克萨斯 | TDPSA | 2024-07 | 无收入门槛，适用范围最广 |
| 蒙大拿 | MCDPA | 2024-10 | 无收入门槛 |
| 俄勒冈 | OCPA | 2024-07 | 含非营利组织 |

**对 looply 的影响**：
- 如果面向全美用户，实际上需要满足最严格州的要求（以 CCPA/CPRA 为基准）
- 德克萨斯和蒙大拿无收入门槛，即使 looply 初期规模小也适用
- **建议做法**：以 CCPA/CPRA 为基准设计，覆盖绝大多数州的要求

## 3. 三方登录合规要求

### 3.1 Google OAuth

| 项目 | 要求 |
|------|------|
| 数据范围 | 只请求必要的 scope（email、profile），不过度获取 |
| 隐私政策 | 必须在 Google Cloud Console 中提供隐私政策 URL |
| 数据使用 | 获取的用户数据只能用于声明的用途 |
| 验证 | 需通过 Google OAuth 验证审核（请求敏感 scope 时） |

### 3.2 Apple Sign In

| 项目 | 要求 |
|------|------|
| 强制要求 | 如果 App 提供任何三方登录，必须同时提供 Apple Sign In（App Store 审核要求） |
| Hide My Email | 用户可选择隐藏真实邮箱，Apple 生成 relay 地址（@privaterelay.appleid.com） |
| 对 looply 的影响 | 必须支持 relay 邮箱收发邮件；不能要求用户提供真实邮箱；relay 邮箱需在 Apple Developer 后台配置域名 |
| 姓名获取 | 用户可选择不提供真实姓名，需容错处理 |

**必须做**：
- 配置邮件域名以支持向 @privaterelay.appleid.com 发送邮件
- 代码中不能假设邮箱格式（relay 邮箱格式特殊）
- 首次授权时获取的 name/email 需立即存储（后续授权不再返回）

### 3.3 Facebook Login

| 项目 | 要求 |
|------|------|
| 数据范围 | 默认只能获取 public_profile 和 email |
| 邮箱可能为空 | 用户可能未在 Facebook 绑定邮箱，或拒绝共享 |
| 数据使用审核 | 需通过 Facebook App Review（请求额外权限时） |
| 对 looply 的影响 | 必须处理 email 为空的情况（允许无邮箱注册，后续引导绑定） |

## 4. 行业惯例（二手电商特有）

### 4.1 服务条款与用户协议

| 级别 | **必须遵守** |
|------|-------------|

- 注册时必须展示并要求用户同意服务条款（Terms of Service）和隐私政策（Privacy Policy）
- 建议使用"点击注册即表示同意"模式（clickwrap），而非仅底部链接（browsewrap）
- 服务条款需包含：用户年龄要求、禁止行为、争议解决机制、责任限制

### 4.2 账号安全行业标准

| 级别 | **建议遵守** |
|------|-------------|

- 登录失败锁定：连续 5 次失败后临时锁定或增加验证
- 异常登录检测：新设备/新地区登录时发送通知或要求额外验证
- 会话超时：长时间不活跃自动登出（建议 30 天 refresh token）
- 密码泄露检测：注册/改密时检查密码是否在已知泄露库中

### 4.3 二手电商特有合规

| 项目 | 要求 | 级别 |
|------|------|------|
| 卖家身份验证 | 首次出售时需验证身份（防欺诈） | 建议遵守 |
| 交易记录保留 | 交易相关数据需保留一定期限（税务/争议） | 必须遵守 |
| 1099-K 报税 | 年交易额 > $600 需向 IRS 报告（2024 起） | 必须遵守（卖家模块） |
| 反洗钱（AML） | 大额交易需 KYC 验证 | 建议遵守（高价值商品） |

**注意**：1099-K 和 AML 主要影响交易模块，但注册模块需预留身份验证字段（如 SSN 后四位、地址），在卖家首次交易时收集。
