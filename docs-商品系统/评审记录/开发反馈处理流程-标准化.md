# 开发反馈处理流程（标准化）

## 流程概览

```
收到开发反馈 → 分类分析 → 逐条给建议 → 用户确认 → 批量执行修改 → 输出反馈记录 → 生成回复草稿 → 更新变更日志
```

---

## 第一步：接收反馈

**输入**：开发提供的问题清单（文字/截图/文件）

**输出**：逐条提取问题，编号

**注意事项**：
- 确认反馈基准版本（如"交付开发 20260420（原型 v14 + PRD v0.2）"）
- 确认当前最新版本（如"原型 v26 + PRD v0.5"）

---

## 第二步：分类与分析

对每条反馈进行分类：

| 分类 | 说明 | 处理方式 |
|------|------|---------|
| **PRD 问题** | PRD 描述不清、有歧义、有遗漏 | 修改 PRD |
| **设计稿问题** | 设计稿和 PRD 不一致、交互缺失 | 修改设计稿 |
| **需求澄清** | 开发对需求理解有疑问，需要产品确认 | 给出明确回复 |
| **技术建议** | 开发提出的技术方案建议 | 评估后决定是否采纳 |
| **版本范围** | 功能不在当前版本范围内 | 说明版本规划 |

---

## 第三步：逐条给出建议

每条反馈输出以下内容：

### 3.1 反馈成立性判断

- **成立**：当前版本需要补充
- **部分成立**：部分已覆盖，部分需要补充
- **不成立**：当前版本已覆盖或不在范围内

### 3.2 当前版本处理方式

说明当前版本如何处理该问题：
- 已覆盖的能力（引用 PRD 具体章节）
- 需要补充的能力（具体方案）
- 不做的能力（说明原因和后续版本规划）

### 3.3 后续行动清单

列出需要执行的具体行动：
- [ ] PRD 补充 XX 章节
- [ ] 原型补充 XX 页面
- [ ] 设计稿修改 XX 部分

### 3.4 建议回复（给开发的）

起草给开发的回复文本，说明：
- 问题是否成立
- 当前版本如何处理
- 后续版本如何覆盖（如有）

---

## 第四步：用户确认（硬性检查点）

**必须等待用户确认后再执行修改**

将所有建议汇总给用户确认：
- 每条标注"同意 / 不同意 / 待讨论"
- 用户确认后再执行修改

**不能自行决定改不改**

---

## 第五步：批量执行修改

按确认结果修改对应文档：

### 5.1 PRD 修改

- **全局约定补充**：在 PRD 开头的全局约定章节补充通用规范
- **章节内容补充**：在对应功能章节补充具体描述
- **版本号更新**：如果是原位修改，不增加版本号；如果是新增章节，更新版本号
- **原型引用更新**：如果原型版本号变化，批量替换全文引用

### 5.2 原型修改

- **新增页面**：按照后台原型规范生成新页面
- **修改页面**：在现有页面上补充内容
- **版本号递增**：v26 → v27

### 5.3 设计稿修改

- 同步更新 PC 端和 APP 端设计稿
- 确保设计稿与 PRD 描述一致

---

## 第六步：输出反馈记录

生成反馈处理记录文件：`~/Desktop/海外业务{模块名}/评审记录/给开发的回复-{日期}.md`

### 记录格式

```markdown
以下为针对本轮反馈的逐条回复，基准版本为 **交付开发 {日期}（原型 v{版本} + PRD v{版本}）**，当前最新版本为 **原型 v{版本} + PRD v{版本}**。

---

### 1. {问题标题}

这个反馈{成立性判断}。

**当前版本的处理方式：**

{具体说明}

**后续版本按需覆盖的能力：**（如有）

{列出后续版本能力}

**后续行动：**
- ✅ {已完成的行动}（已完成：{具体位置}）
- [ ] {待完成的行动}

---

### 总结

| # | 问题 | 处理方式 |
|---|------|---------|
| 1 | {问题} | {处理方式} |
| 2 | {问题} | {处理方式} |

**后续行动：**

1. PRD 补充（问题 X、Y）：{具体内容}
2. 原型补充（问题 Z）：{具体内容}

如后续开发仍需基于最新版逐项对齐，我这边继续以 **v{版本} 原型 + PRD v{版本}** 作为唯一基准版本配合确认。
```

---

## 第七步：生成回复草稿

生成给开发的回复文本，逐条说明处理结果：

```markdown
感谢反馈，已逐条处理如下：

1. {问题}：{处理结果}
2. {问题}：{处理结果}

已更新文档：
- PRD v{版本}：{更新内容}
- 原型 v{版本}：{更新内容}

最新文档位置：
- PRD：~/Desktop/海外业务{模块名}/PRD/looply-{模块名}-PRD-v{版本}.md
- 原型：~/Desktop/海外业务{模块名}/原型/looply-{模块名}后台原型-v{版本}.html

如有其他问题，随时沟通。
```

用户确认后发送。

---

## 第八步：更新变更日志

更新 `~/Desktop/海外业务{模块名}/looply-{模块名}-变更日志.md`

### 8.1 更新"开发反馈处理"条目

在"变更内容"中追加本次改动：
- 补充全局约定：{具体内容}
- {章节名}章节新增/更新：{具体内容}
- 全文原型文件引用更新为 v{版本}

### 8.2 追加迭代记录

在"迭代记录"中追加：
- {日期}：{本次改动摘要}

### 8.3 更新影响范围

更新"影响范围"章节：
- **原型**：v{旧版本} → v{新版本}，{改动内容}
- **PRD**：v{版本} 原位更新（{改动内容}）
- **实体关系图**：{是否需要同步更新}

---

## 自检清单

完成后逐项检查：

- [ ] 每条反馈都有分类和处理建议
- [ ] 需要修改的文档都已标注
- [ ] 用户确认后才执行修改（不能自行决定改不改）
- [ ] PRD 修改后版本号已更新（如需要）
- [ ] 原型修改后版本号已递增
- [ ] 设计稿与 PRD 描述一致
- [ ] 反馈记录已存档
- [ ] 回复草稿已生成
- [ ] 变更日志已更新
- [ ] 所有"后续行动"已标记完成状态（✅）

---

## 反模式（禁止）

1. **不分类直接改**：必须先分类、给建议、用户确认后再改
2. **只改 PRD 不改设计稿**：涉及设计稿的问题必须同步修改
3. **替用户做决定**：需求澄清类问题必须让用户确认
4. **不更新版本号**：原型修改后必须更新版本号
5. **不存档**：每次反馈处理都要有记录
6. **不更新变更日志**：每次改动都要记录到变更日志

---

## 关键原则

1. **用户确认优先**：不确定的需求先跟用户确认，不要自己假设
2. **文档同步更新**：PRD、原型、设计稿、变更日志必须同步更新
3. **版本号管理**：原型修改必须递增版本号，PRD 原位修改不增加版本号
4. **完整性检查**：每次修改后全量检查相关章节是否完整
5. **可追溯性**：所有改动都要记录到变更日志和反馈记录中

## 附录：停用级联规则补充规范

当补充停用级联规则时，遵循以下规范：

### 规范内容

在对应实体的编辑章节中，新增"异常处理"段落：

```markdown
**异常处理**

- 停用{实体名}时有关联数据：弹窗展示影响范围预览列表（{受影响实体}数及{实体名称}列表），用户确认后执行停用
```

### 已覆盖的实体

- ✅ 类目：关联品牌 + SPU
- ✅ 品牌：关联系列 + SPU
- ✅ 系列：挂载 SPU
- ✅ 属性分组：该分组下的属性定义
- ✅ 属性定义：引用该属性的 SPU + SKU
- ✅ 渠道：在售商品（自动下架）

### 对齐全局约定

所有停用级联规则都对齐全局约定（第 121 行）中的影响范围预览规范：
- 展示受影响实体的数量和名称列表
- 最多展示 10 条，超出显示"等 N 个{实体名称}"
- 预览列表支持滚动查看完整数据
- 用户确认后执行停用操作
