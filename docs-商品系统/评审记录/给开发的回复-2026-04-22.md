以下为针对本轮反馈的逐条回复，基准版本为 **交付开发 20260420（原型 v14 + PRD v0.2）**，当前最新版本为 **原型 v22 + PRD v0.4**。

### 1. 关于「SKU 批量生成」

该功能**已在交付范围内**。

PRD v0.2 第 **2.10.2 SPU 新建**中已定义「SKU 批量生成区域」：运营在 SPU 新建/编辑页选择销售属性启用值后，系统自动生成 SKU 矩阵（笛卡尔积），并可在矩阵中直接维护 SKU 编码、货号、参考价等信息。

原型 v14 的 SPU 新建页也已有对应区域，因此当前方案并不是要求运营逐个进入 SKU 新建页手工录入。

如后续讨论的是更高级的批量能力（如 Excel 导入、跨 SPU 批量生成、批量改价等），这类能力不在本次交付范围内，可作为后续版本需求单独规划。

### 2. 关于 `pending_review` / 审核流程缺失

这一点反馈成立，属于**交付版清理不彻底导致的歧义**。

PRD v0.2 在 **1.2 不做什么** 中已明确说明：
- 商品审核流程不在当前版本范围内

但交付版原型 v14 中仍残留了：
- 「提交审核」按钮
- 「待审核」状态
- 审核相关文案

这会让人误以为当前版本包含完整审核链路。该问题已在最新版中统一修正：

- 原型 v22 已移除审核相关入口与文案
- PRD v0.4 已统一改为 **待核验**（指质检/验收语义，不是审批流）
- 当前版本不存在审核人视图、审核意见、驳回原因、审核历史等审批页面

因此，**请以最新版（v22 + PRD v0.4）为准**。当前版本不包含商品审核流程。

### 3. 关于快捷上架缺少规则定价能力

这部分属于**后续版本能力**，不在当前 MVP 范围内。

你提到的以下能力：
- 基于基准价 ±N%
- 同步某渠道价格
- 比对竞品价位
- 多渠道规则化调价

方向是合理的，但 PRD 中已明确排除高级定价能力。当前版本仅支持：
- 快捷模式：统一挂牌价
- 自定义模式：按渠道分别填写挂牌价

规则定价、多币种定价等能力已放在后续版本规划中处理。当前版本以先打通基础上架能力为目标。

### 4. 关于列表页筛选器数量 / 默认排序 / 列表规范

这里拆分来看：

#### 4.1 筛选器数量不一致
这部分**不完全视为问题**。

不同列表页的数据结构和使用目标不同，因此筛选维度允许不同：
- 类目管理更偏树形结构检索，重点是父类目
- 品牌管理更偏状态与名称检索
- 商品管理涉及成色、状态、鉴定状态等更多业务维度

所以筛选器数量不要求机械一致。

#### 4.2 默认排序未明确
这一点反馈成立，已补充。

当前已在 **PRD v0.4 全局约定** 中补充：
- 所有列表页默认按**创建时间倒序**展示

#### 4.3 空值显示 / 列表基础规范
当前已补充：
- 空值统一显示为 **“—”**

后续如进入后台组件化阶段，会再进一步沉淀列表页模式库；当前版本先补齐基础约定，保证实现口径一致。

### 5. 关于颜色 / 材质等视觉属性缺少富数据

这个建议方向认可，但**不纳入当前版本补丁范围**。

原因是它不是单点 UI 优化，而是会影响整套属性数据结构和展示链路，包括但不限于：
- 属性选项数据模型扩展（如 color_code / icon_url / preview_image）
- 后台属性选项管理页
- SPU / SKU 录入选择器
- 前台商品展示页

因此这更适合作为一个完整专题，在后续版本中与前台展示能力一起规划。

当前版本建议先保持文本值方案，但命名上尽量清晰，例如：
- Black / 黑色
- Ivory / 象牙白
- Amber Gold / 琥珀金

### 6. 关于列表页空态缺失

该反馈成立，且已按**通用方案**处理，不只针对商品模块。

当前已统一补齐后台原型空态规范，并沉淀为可复用标准：

#### 已定义的 3 类标准空态
1. **零数据引导态**
   - 适用于首次使用、列表尚无任何数据
   - 展示引导文案 + CTA（如“新增第一个类目”）

2. **筛选无匹配态**
   - 适用于有数据但当前搜索/筛选无结果
   - 展示提示文案 + “清除筛选条件”按钮

3. **网络异常态**
   - 适用于接口加载失败
   - 展示错误提示 + “重试”按钮

#### 处理结果
已完成以下同步：
- 写入 `looply-admin-ui-spec.md`，作为后台模块通用 UI 规范
- 写入 PRD v0.4 全局约定
- 在原型生成脚本中封装为空态模板函数，后续各模块可直接复用

说明：**权限不足态**暂未纳入当前版本，因为当前版本尚未进入权限模块设计阶段，先预留规范位置。

### 7. 关于质检能力边界（QC / 质检报告 / 多轮质检）

这个反馈方向成立，但需要明确边界：**商品域当前只记录质检结果，不承载质检过程**。

当前版本在商品管理中保留了基础质检结果录入与查看能力，用于支撑商品信息沉淀和内部运营判断；但以下能力不在商品域当前范围内：
- 多轮质检（初检 / 复检 / 终检）
- 质检员归属 / 责任记录
- 质检任务流转
- 对外质检报告页

后续会独立建设**质检域**，由质检域承接完整质检流程；商品域只接收并保存最终的质检结果快照。

另外，商品域中的质检结果会按统一结构沉淀，当前方向为：
- 按后台类目配置质检模板
- 结果结构为：**维度 → 质检项 → 质检结果**
- 例如：**外观 → 划痕 → 轻微**

因此，这一条不按“当前版本缺完整质检流程”处理，而是按“当前版本边界需要明确”处理。

### 8. 关于 Brand 国别代码 / 国家主数据

这个反馈成立，当前版本已明确边界。

商品域当前**不独立建设国家主数据模块**，品牌和渠道等场景中的国家字段先按**受控国家枚举**处理，不要求运营手工记忆或录入 ISO code。

后续会由**独立国家主数据模块**统一提供国家数据（含扩展和接口支持），商品域通过接口引用，不在商品域内部重复建设国家管理页面。

也就是说：
- 当前版本：国家 = 受控枚举
- 后续版本：国家主数据 = 独立模块提供

### 9. 关于多角色协同 / 权限粒度

这个反馈成立，但属于**当前版本不展开的范围边界**，不是本轮商品域单独补齐的内容。

当前版本页面默认按“**具备对应模块权限的后台用户**”设计，不在商品域内单独展开：
- 鉴定师
- 质检员
- 定价专员
- 库存管理员
- 销售运营
- 审核员
等角色的差异化权限视图。

后续如进入团队化协作阶段，会由**独立权限/账号体系**统一建设角色、权限矩阵、只读/可编辑差异、审批权限和审计追责机制，商品域再按统一权限体系对接。

因此，这一条按“后续由独立权限体系承接”处理。

### 10. 关于 warehouse / 仓库字段

这个反馈成立，且已按最新方案处理。

由于当前版本**没有仓库主数据模块**，也没有仓储视图、库存地点管理、调拨流程等完整能力，因此继续保留商品上的“归属仓库/门店”字段会形成悬空实体。

最新版处理中，已从商品域 PRD 中删除相关字段，避免在没有仓库域支撑的情况下落一个孤立字段。

后续如果正式建设仓库域，再重新引入：
- 仓库 / 门店主数据
- 库存地点视图
- 调拨流程
- 商品与仓库归属关系

因此，这一项不是继续保留并让开发自行兜底，而是**当前先删除，后续由独立仓库域承接**。

### 11. 关于 AttrDefinition 规则过多、关系不清

这个反馈成立，当前已按简化方案处理。

已做两项调整：

1. **移除“关键属性”**
   当前属性类型统一收敛为：
   - **销售属性**
   - **描述属性**

2. **补充属性配置规则矩阵**
   已在 PRD v0.4 中明确：
   - 属性类型 × 值类型 × 录入方式的允许组合
   - 销售属性与 SKU 生成关系
   - 数值型属性与单位支持关系
   - 枚举型属性与属性选项管理的依赖关系
   - 当前版本中哪些组合禁止使用

当前目标是先降低运营配置理解成本，避免由于字段过多但规则不清造成培训和实施偏差。
