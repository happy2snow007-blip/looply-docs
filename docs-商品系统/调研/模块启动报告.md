# 商品系统模块启动报告

## 模块定位与目标

- **模块**：商品系统（商品域）
- **业务背景**：looply 二手电商平台，B2C 模式（三方商家 + 自营），前期主攻美国市场
- **品类结构**：前期奢侈品占比 ≥80%，消费电子约 20%
- **核心目标**：商品能发、能看、能买，合规底线达标，数据架构保证多国扩展性

## 系统边界

### 系统定位

商品系统是 looply 平台的商品生命周期管理中枢，负责商品从发布到可售的全流程。

核心职责：
- 商品信息的结构化存储与管理
- 商品质量标准的定义与执行（成色分级、鉴定、召回拦截）
- 商品可售性的判断与控制（审核、库存、状态）
- 商品价格体系的管理（基础定价、渠道差异定价）

### 边界判断原则

**归属商品系统的判断标准：**
1. 数据主体是"商品"本身的属性或状态
2. 逻辑围绕"这个商品能不能卖、怎么卖"
3. 变更频率与商品生命周期强相关

**不归属商品系统的判断标准：**
1. 数据主体是"用户""订单""支付"等其他领域实体
2. 逻辑围绕"谁买了、怎么付钱、怎么履约"
3. 跨多个商品的聚合计算或推荐排序

### 明确归属

| 能力 | 归属 | 理由 |
|------|------|------|
| 商品基础信息（标题/描述/图片/品类/品牌） | ✅ 商品系统 | 商品核心属性 |
| 成色分级与质检描述 | ✅ 商品系统 | 二手商品核心质量标准 |
| 奢侈品鉴定流程与证书 | ✅ 商品系统 | 商品可售性前置条件 |
| CPSC 召回拦截 | ✅ 商品系统 | 商品上架合规校验 |
| 商品审核流程 | ✅ 商品系统（MVP 内置） | 商品状态机的一部分，后期可拆分为独立审核系统 |
| 库存管理 | ✅ 商品系统（MVP 简化版） | MVP 支持流程闭环，长期拆分为独立库存系统 |
| 商品基础售价 | ✅ 商品系统 | 商家对商品的定价 |
| 渠道差异定价 | ✅ 商品系统 | 同一商品在不同渠道的价格策略 |
| 运费策略 | ✅ 商品系统（MVP 简化版） | MVP 内置基础运费规则，长期拆分为独立运费系统 |
| 商品税务分类码 | ✅ 商品系统 | 商品合规属性，始终归商品系统 |
| 税费计算 | ❌ 支付结算系统（MVP）→ 独立税费系统（长期） | 基于订单金额和地址的实时计算，商品系统只提供税务分类码 |
| 平台佣金费率 | ❌ 支付结算系统 | 平台收入策略，非商品属性 |
| 促销折扣价 | ❌ 营销系统 | 跨商品的营销活动，非单品属性 |
| 商家信息展示 | ❌ 商家系统 | 商家域数据，商品系统只存 merchant_id 关联 |
| 假冒品举报处理 | ❌ 客服/合规系统 | 用户投诉处理流程，商品系统只提供举报入口 |
| 订单创建与支付 | ❌ 订单系统 + 支付系统 | 交易流程，商品系统只提供商品信息和库存扣减接口 |
| 商品搜索与排序 | ❌ 搜索系统 | 跨商品的检索与推荐，商品系统只负责数据同步 |

### MVP 内置能力的拆分预期

以下能力 MVP 阶段内置于商品系统，长期需拆分为独立系统：

| 能力 | MVP 阶段 | 长期目标 | 商品系统保留的角色 |
|------|---------|---------|------------------|
| 商品审核 | 审核规则硬编码在商品系统内 | 可拆分为独立审核系统 | 提供审核状态机，对接外部审核规则引擎 |
| 库存管理 | 基础库存字段（总库存/可售库存/扣减接口） | 独立库存系统（支持预占、多渠道分配） | 只存商品维度的库存快照，实时库存由库存系统管理 |
| 运费策略 | 基础运费规则（按重量/按件） | 独立运费系统（支持区域定价、包邮策略、物流商对接） | 只存商品物流属性（重量/尺寸），运费计算交运费系统 |

### 对外接口约定

**商品系统提供的接口：**
- 商品信息查询（详情、列表、批量查询）
- 库存扣减（原子操作，支持回滚）
- 商品状态变更通知（上架/下架/删除，推送给搜索系统）
- 可售性校验（库存充足 + 状态已上架 + 未召回）

**商品系统依赖的接口：**
- 支付结算系统：查询佣金费率（用于商家端预估收益展示）
- 商家系统：查询商家信息（用于商品详情页展示）
- 风控系统：商品审核规则校验（图片鉴黄、文本敏感词）
- 鉴定服务商：奢侈品真伪鉴定（Legit Check / Real Authentication）
