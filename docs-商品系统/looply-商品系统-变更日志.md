##### 🔄 派生维度 + 属性值映射管理（迭代中）
- **原型版本**：v14 → v15 → v20
- **原型文件**：looply-商品管理后台原型-v20-风格A-活力亲和.html
- **文件位置**：~/Desktop/海外业务/商品/原型/
- **状态说明**：功能还在迭代调整中，暂不交付开发
- **设计方案**：派生维度方案（方案B）
  - 派生维度是独立实体，与客观属性完全分离
  - 派生维度值通过映射规则从客观属性实时计算，不存储在商品上
  - 一个源属性选项可映射到多个派生维度选项（多对多关系）
  - 映射规则变更全量实时生效，无需重新计算商品数据

- **新增内容**：
  - 新增菜单项："派生维度"（位于"属性管理"分组下，"属性映射"后）
  - 新增 7 个页面：
    - **派生维度管理**（4 个页面）：
      - 派生维度列表页（page-derived-dimension-list）
      - 派生维度新建页（page-derived-dimension-create）
      - 派生维度编辑页（page-derived-dimension-edit）
      - 派生维度详情页（page-derived-dimension-detail）
    - **属性值映射管理**（3 个页面，v15→v16 重构）：
      - 属性值映射列表页（page-attr-mapping-list）
      - 属性值映射新建页（page-attr-mapping-create）
      - 属性值映射编辑页（page-attr-mapping-edit）

- **功能说明**：
  - **派生维度**：从客观属性计算得出的业务概念属性（如从"颜色"派生"色系"）
  - **属性值映射**：配置属性选项到派生维度选项的映射关系（如"深空黑"→"黑色系"）
  - **典型场景**：颜色→色系、屏幕尺寸→尺寸分组、存储容量→容量分组
  - **用途**：前端筛选、营销选品（用户按"色系"筛选时，通过映射表查询所有映射到该色系的具体颜色）

- **页面功能**：
  - **派生维度列表页**：展示所有派生维度，KPI 统计（派生维度数/启用维度/派生选项总数/映射关系总数）
  - **派生维度新建页**：3 步向导流程
    - Step 1：基础信息（维度名称、来源属性、状态）
    - Step 2：配置派生维度选项（选项值、排序、状态）
    - Step 3：预览
  - **派生维度编辑页**：编辑维度信息、管理选项、查看关联映射统计
  - **派生维度详情页**：查看完整信息和关联的属性值映射关系
  - **属性值映射列表页**：展示映射关系列表（删除"映射组"概念）
  - **属性值映射新建页**：3 步向导流程
    - Step 1：选择源属性和派生维度
    - Step 2：配置映射关系（左侧选源选项，右侧选派生维度选项，建立关联）
    - Step 3：映射关系预览表格
  - **属性值映射编辑页**：调整已有映射关系

## 使用说明

### 版本号规则
- 小版本号递增（V1.0 → V1.1 → V1.2）：功能调整、页面优化、字段变更
- 大版本号递增（V1.0 → V2.0）：架构重构、大功能模块新增

### 功能成熟度标记
每个功能模块用标记区分当前状态：
- ✅ **稳定**：已定稿，可以开始开发
- 🔄 **迭代中**：还在调整，暂时不要动
- 📝 **草稿**：刚开始设计，变化很大

### 变更记录格式

#### 迭代过程中
- 每个功能模块独立记录，标注成熟度状态
- 迭代中的功能：在"迭代记录"下追加每次调整（记增量）
- 稳定的功能：记录最终方案（全量描述）

#### 功能定稿时
- 把迭代记录合并成"最终方案"（开发只需看最终结果）
- 迭代历史折叠保留，标注"开发可跳过"
- 状态从 🔄 改为 ✅，更新"状态说明"

#### 冲突处理
当迭代中的功能影响了已稳定的功能时：
1. 方案 A：把稳定功能降级回 🔄，标注原因
2. 方案 B：在稳定功能下追加"待调整"记录，标注依赖关系

示例：
```markdown
##### ✅ 属性管理列表页（稳定，待调整）
- 状态说明：已定稿，但依赖"属性映射"功能定稿后需追加一列
- 待调整：新增"已关联映射数"列（依赖属性映射功能定稿）
```

### 交付流程
1. 在"进行中"版本下持续记录变更
2. 只有标记为 ✅ 稳定的功能才能交付开发
3. 攒到合理批次后，标记"已交付"并更新交付日期
4. 将最新文件复制到 `~/Desktop/海外业务/商品/交付开发/` 文件夹
5. 交付包命名规范：`交付开发 V{版本号}-{日期}`（如 `交付开发 V1.0-20260420`、`交付开发 V1.1-20260425`），版本号与变更日志对齐
6. 同步到文档中心（looply-docs）
7. 通知开发团队查看变更日志

### 开发视角
开发只需关注：
- 从上次交付版本到当前版本的所有"已交付"条目
- 只看标记为 ✅ 稳定的功能（🔄 迭代中的功能暂时不动）
- 每条变更的"影响范围"（哪些实体/接口/页面受影响）
- 是否需要调整已写的代码
