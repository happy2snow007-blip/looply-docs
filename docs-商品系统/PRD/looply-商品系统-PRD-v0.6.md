# looply 商品系统 PRD V0.6

> 版本：V0.6 | 日期：2026-04-24 | 范围：基础数据关系 + 属性管理 + 属性作用域 + 标品管理（SPU/SKU）+ 质检报告模板 + 实物商品 + 渠道管理 + 渠道商品 + 标签管理 + 尺码体系 + 品牌尺码体系

## 一、概述

### 1.1 背景与目标

looply 是一个面向海外市场的二手电商平台，采用 B2C 模式（三方商家 + 自营），前期主攻美国市场，品类以奢侈品为主（占比 ≥80%），消费电子约 20%。

商品系统是平台的商品生命周期管理中枢，负责商品从发布到可售的全流程。本 PRD 当前版本覆盖商品系统的基础数据层，包括：

- 基础数据关系：类目、品牌、系列及其关联关系的管理
- 属性管理：属性定义、属性选项、度量单位的管理
- 属性作用域：按类目/品牌/系列三层维度控制属性的启用、必填、可选值
- 标品管理：SPU（标准产品单元）和标品 SKU 的创建与维护
- 实物商品管理：一物一码的实物商品录入、质检、鉴定和库存管理
- 渠道管理：销售渠道的创建与配置
- 渠道商品：实物商品在各渠道的上架管理
- 标签管理：商品标签的创建与分类管理
- 尺码体系：标准尺码体系管理、尺码换算、品牌尺码体系及品牌尺码换算

### 1.2 不做什么

- 前台类目管理（前台展示类目树及前后台类目映射，后续版本覆盖）
- 成色质检、鉴定安全、合规字段等独立模块（当前版本中质检和鉴定作为二手商品录入的子流程内嵌）
- 渠道定价规则（促销价、折扣策略、自动调价等高级定价能力，已从原型中移除；基础挂牌价在渠道商品中支持）
- 商品发布流程、商品审核流程（已从原型中移除）

### 1.3 用户角色

| 角色 | 说明 |
|------|------|
| 运营人员 | 管理类目、品牌、系列、属性体系，配置属性作用域规则 |

> 当前版本仅覆盖运营后台，商家端、质检端后续版本覆盖。

### 1.4 核心场景

1. 运营人员搭建品类体系：创建多层级类目树，建立品牌库和系列库，配置类目-品牌-系列的关联关系
2. 运营人员定义属性体系：定义属性及其选项，配置度量单位
3. 运营人员管理尺码体系：创建标准尺码体系（如 IT、FR、US），维护尺码值和跨体系换算关系
4. 运营人员管理品牌尺码体系：创建品牌专属尺码体系（如 Moncler 0-6 码），建立品牌尺码到标准尺码的映射
5. 运营人员配置属性作用域：针对不同类目/品牌/系列组合，控制哪些属性启用、是否必填、可选哪些值
6. 运营人员创建标品 SPU：选择类目/品牌/系列，填写 SPU 信息和属性值，上传标品图片
7. 运营人员创建标品 SKU：基于 SPU 的销售属性组合生成 SKU 矩阵，维护 SKU 编码、货号和参考价
8. 运营人员录入实物商品：关联标品 SKU → 填写实物信息 → 录入质检和鉴定 → 设定价格，4 步完成商品录入
9. 运营人员管理渠道商品：查看实物商品在各渠道的上架状态，执行上架/下架操作
10. 运营人员管理渠道：创建和配置销售渠道（线上自营、线下门店、三方平台）
11. 运营人员管理标签：创建和维护商品标签，按适用对象分类（实物商品/渠道商品/通用）

### 1.6 术语说明

| 术语 | 说明 |
|------|------|
| 后台类目 | 商品主数据的分类维度，用于属性作用域配置和数据管理，不直接面向消费者 |
| 品牌 | 商品所属品牌，与类目为多对多关系 |
| 系列 | 品牌下的产品系列（如爱马仕的 Birkin、Kelly），与类目为多对多关系 |
| 属性定义 | 单个属性的元数据（名称、值类型、录入方式等） |
| 属性选项 | 枚举型属性的全局候选值 |
| 度量维度 | 度量单位的分类（如长度、重量、容量） |
| 度量单位 | 具体的计量单位（如 mm、cm、inch），归属于某个维度 |
| 属性作用域 | 控制属性在特定类目/品牌/系列组合下的启用状态、必填性、可选值范围 |
| 作用域层级 | 三层优先级：CATEGORY_SERIES > CATEGORY_BRAND > CATEGORY |
| SPU | 标准产品单元（Standard Product Unit），代表一个标准商品（如 Hermes Birkin 25），不区分具体规格 |
| 标品 SKU | 标准规格单元（Standard SKU），SPU 下按销售属性组合（如颜色×材质）拆分的具体规格 |
| 销售属性 | 影响 SKU 生成的规格属性（如颜色、材质），不同销售属性值的组合对应不同 SKU |
| 描述属性 | 补充描述信息的属性（如工艺、产地），不影响 SKU 拆分 |
| 实物商品 | 一物一码的实物商品（product），每件实物绑定一个标品 SKU，有独立的成色、质检、鉴定和库存状态 |
| 渠道 | 销售渠道（如线上自营商城、线下门店、亚马逊等三方平台） |
| 渠道商品 | 一件实物商品在某个渠道的上架记录（listing），同一实物可在多个渠道同时上架 |
| 成色等级 | 实物商品的外观状态分级（A 几乎全新 / B 轻微使用痕迹 / C 明显使用痕迹） |
| 标签 | 商品标签，用于分类和筛选，按适用对象分为实物商品标签、渠道商品标签和通用标签 |
| 尺码体系 | 标准尺码体系（如 IT 意大利尺码、FR 法国尺码、SHOE_EU 欧洲鞋码），定义一组有序的尺码值 |
| 尺码换算 | 不同标准尺码体系之间的对照关系（如 IT 38 = FR 34 = UK 6） |
| 品牌尺码体系 | 品牌专属的尺码体系（如 Moncler 0/1/2/3/4/5/6 码），独立于标准尺码体系 |
| 品牌尺码换算 | 品牌尺码到标准尺码的映射关系（如 Moncler 0 = IT 38）

### 1.7 多语言 / 多国家策略

- 当前版本仅支持美国市场，界面语言为英文
- 类目名称、品牌名称、系列名称等字段预留中英文双字段（如 brand_name / brand_name_en）
- 度量单位已预留国家展示单位机制（country_display_unit），当前版本不启用前端交互，仅做数据结构预留

## 二、需求详细描述

> **全局约定**
>
> - **前置条件**：以下所有功能均要求用户已登录商品管理后台且具有对应模块的操作权限。各功能项不再逐个标注。当前版本不展开角色权限矩阵，页面默认按"具备对应模块权限的后台用户"设计；多角色协同、最小权限控制、审批角色拆分由后续独立权限体系建设。
> - **默认排序规则**：所有列表页默认按创建时间倒序排列（最新创建的记录排在最前）。如有特殊排序需求，在对应功能项中单独说明。
> - **空值显示规范**：表格中的空值统一显示为"—"（中文破折号），不留空白。布尔型字段显示对应状态标签（如"启用/停用"），不显示 true/false。
> - **外部主数据引用规则**：当前 PRD 中如引用国家、权限、质检、仓库等外部基础域能力，若未在本模块中展开对应主数据或流程页面，则默认表示"引用外部模块能力"，不在商品域内独立建设。当前版本中的国家数据先采用受控枚举，后续由独立国家主数据模块统一提供接口支持。
> - **操作记录规范**：所有编辑页和详情页均包含"操作记录"区域，记录该实体的操作历史。操作记录包含以下字段：操作时间、操作人、操作类型（创建/编辑/启用/停用/上架/下架/状态变更等）、操作详情（具体变更内容，如"修改基准售价：$100 → $120"）。操作记录按时间倒序展示，最新操作在最上方。各功能项不再逐个标注操作记录区域，默认包含。
> - **影响范围预览规范**：停用操作（类目/品牌/系列/渠道/标签等）如涉及关联数据，系统展示影响范围预览列表，包含受影响实体的数量和名称列表（最多展示 10 条，超出显示"等 N 个{实体名称}"），预览列表支持滚动查看完整数据。用户确认后执行停用操作。
> - **页面状态变体**：所有表单页支持加载态（数据请求中显示加载指示器）、提交中置灰态（防止重复提交）、字段校验错误态（错误提示显示在对应字段下方）。所有列表页支持以下状态：
>   - **加载态**：数据请求中显示加载指示器
>   - **零数据引导态**：列表无任何数据时，隐藏搜索/筛选栏，展示居中引导区（图标 + 标题"暂无{实体名称}" + 描述"点击下方按钮创建第一个{实体名称}" + 主按钮"新增{实体名称}"）
>   - **筛选无匹配态**：搜索或筛选后无结果时，保留搜索/筛选栏，表格区域替换为居中提示（图标 + 标题"未找到匹配结果" + 描述"请尝试调整搜索关键词或筛选条件" + 按钮"清除筛选条件"）
>   - **网络异常态**：请求失败时，保留搜索/筛选栏，表格区域替换为居中提示（错误图标 + 标题"加载失败" + 描述"网络异常，请检查网络连接后重试" + 主按钮"重试"）
>   - 空态组件的视觉规范详见《Looply 后台原型 UI 规范》6.13 节
>   - 特殊状态变体在各功能项中单独说明。

### 2.1 类目管理

#### 模块概述

类目是商品主数据的根维度，采用多层级树形结构（支持 3 级及以上）。类目决定了商品可关联的品牌、系列和属性作用域规则。

#### 2.1.1 类目列表

**功能描述**

展示所有后台类目的扁平列表，支持搜索和筛选，便于运营人员快速定位和管理类目。

**页面元素**

| 元素 | 说明 |
|------|------|
| 统计卡片 | 一级类目数、二三级类目数、关联属性数、挂载 SPU 数 |
| 搜索栏 | 支持按类目 ID / 类目名称搜索 |
| 筛选 | 按父类目筛选 |
| 列表字段 | 类目 ID、类目名称、父类目、层级、状态、覆盖品牌数、覆盖系列数、操作 |
| 操作按钮 | 新增类目、编辑 |

**交互说明**

- 点击"新增类目"跳转类目新建页
- 点击"编辑"跳转类目编辑页

**UI 关联**

- 原型：looply-商品管理后台原型-v28-风格A-活力亲和.html → 类目管理（page-category-list）

#### 2.1.2 类目新建

**功能描述**

新建一个后台类目，设置基础信息。

**页面元素**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 类目名称 | 文本输入 | 是 | 类目的显示名称 |
| 父类目 | 下拉选择 | 否 | 不选则为一级类目 |
| 层级 | 自动计算 | — | 根据父类目自动确定，只读 |
| 排序 | 数字输入 | 否 | 数字越小越靠前 |
| 状态 | 下拉选择 | 是 | 启用 / 停用，默认启用 |

**校验规则**

| 字段 | 规则 | 校验时机 |
|------|------|---------|
| 类目名称 | 不能为空；同一父类目下名称不能重复 | 提交时 |

**操作流程**

1. 运营人员点击"新增类目"进入新建页
2. 填写类目名称，选择父类目（可选）
3. 系统自动计算层级
4. 点击"保存"，校验通过后创建类目，返回列表页

**异常处理**

- 类目名称重复：提示"同一父类目下已存在同名类目"
- 网络异常：提示"保存失败，请重试"

**UI 关联**

- 原型：looply-商品管理后台原型-v28-风格A-活力亲和.html → 类目新建（page-category-create）

#### 2.1.3 类目编辑

**功能描述**

编辑已有类目的基础信息，查看该类目的影响范围（关联品牌、系列、属性、SPU）。

**页面元素**

基础信息区域同新建页，额外增加：

| 字段 | 类型 | 说明 |
|------|------|------|
| 类目 ID | 只读 | 系统生成，不可修改 |
| 更新时间 | 只读 | 最近一次修改时间 |

影响范围摘要：

| 字段 | 说明 |
|------|------|
| 覆盖品牌数 | 当前类目关联的品牌数量 |
| 覆盖系列数 | 当前类目关联的系列数量 |
| 关联属性数 | 当前类目下的属性定义数量 |
| 挂载 SPU 数 | 当前类目下的 SPU 数量 |

已关联品牌列表（只读）：品牌 ID、品牌名称、英文名、关联系列数、状态

已关联系列列表（只读）：系列 ID、系列名称、所属品牌、状态

**交互说明**

- 已关联品牌和系列为只读展示，如需调整需前往品牌管理或系列管理
- 停用类目时，如果该类目下有启用状态的 SPU，需提示确认

**异常处理**

- 修改父类目导致循环引用：提示"不能将类目设为自身或其子类目的子类目"
- 停用类目时有关联数据：弹窗展示影响范围预览列表（关联品牌数及品牌名称列表、挂载 SPU 数及 SPU 名称列表），用户确认后执行停用

**UI 关联**

- 原型：looply-商品管理后台原型-v28-风格A-活力亲和.html → 类目编辑（page-category-edit）

### 2.2 品牌管理

#### 模块概述

品牌是商品的重要维度，与类目为多对多关系（通过 category_brand 关联）。品牌下可创建多个系列。

#### 2.2.1 品牌列表

**功能描述**

展示所有品牌，支持搜索和筛选。

**页面元素**

| 元素 | 说明 |
|------|------|
| 统计卡片 | 品牌总数、启用品牌数、覆盖类目数、关联系列数 |
| 搜索栏 | 支持按品牌 ID / 品牌名称 / 英文名搜索 |
| 筛选 | 按状态筛选 |
| 列表字段 | 品牌 ID、品牌名称、英文名、状态、覆盖类目数、关联系列数、更新时间、操作 |
| 操作按钮 | 新增品牌、编辑 |

**UI 关联**

- 原型：looply-商品管理后台原型-v28-风格A-活力亲和.html → 品牌管理（page-brand-list）

#### 2.2.2 品牌新建

**功能描述**

新建品牌，设置基础信息和覆盖类目。

**页面元素**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 品牌名称 | 文本输入 | 是 | 品牌中文名 |
| 英文名 | 文本输入 | 是 | 品牌英文名 |
| 品牌 Logo | 图片上传 | 否 | 品牌标识图片 |
| 所属国家 | 下拉选择 | 否 | 从受控国家枚举中选择品牌所属国家（法国、意大利、美国、英国、瑞士、日本等）；当前版本不在商品域独立维护国家主数据，后续由独立国家主数据模块提供接口支持 |
| 成立年份 | 数字输入 | 否 | 如 1837 |
| 创始人 | 文本输入 | 否 | 创始人姓名 |
| 品牌介绍 | 多行文本 | 否 | 品牌历史、定位等（内部参考） |
| 品牌描述 | 多行文本 | 否 | 面向消费者的简短描述 |
| 状态 | 下拉选择 | 是 | 启用 / 停用，默认启用 |

覆盖类目区域：

- 点击"添加类目"从类目树中选择要覆盖的类目
- 已添加的类目以列表展示，支持移除

**校验规则**

| 字段 | 规则 | 校验时机 |
|------|------|---------|
| 品牌名称 | 不能为空；全局唯一 | 提交时 |
| 英文名 | 不能为空；全局唯一 | 提交时 |

**操作流程**

1. 填写品牌基础信息
2. 添加覆盖类目（至少一个）
3. 点击"保存"

**异常处理**

- 品牌名称或英文名重复：提示"该品牌名称已存在"

**UI 关联**

- 原型：looply-商品管理后台原型-v28-风格A-活力亲和.html → 品牌新建（page-brand-create）

#### 2.2.3 品牌编辑

**功能描述**

### 2.8 属性作用域

#### 模块概述

属性作用域是商品属性体系的核心配置能力，用于控制属性在不同类目/品牌/系列组合下的启用状态、必填性、可选值范围和可用单位。

采用三层作用域模型，优先级从高到低：

| 层级 | 作用域类型 | 含义 | 示例 |
|------|-----------|------|------|
| 1（最高） | CATEGORY_SERIES | 类目 + 品牌 + 系列 | 手提包 × 爱马仕 × Birkin |
| 2 | CATEGORY_BRAND | 类目 + 品牌 | 手提包 × 爱马仕 |
| 3（最低） | CATEGORY | 仅类目 | 手提包 |

**解析规则**：

- 属性启用判断：按优先级查找第一个 status=1 的作用域配置。三层都没有 status=1 时，该属性不出现在商品配置中
- 属性选项判断：命中某一层则整体使用该层的可用选项集合，不跨层合并。三层都没有该属性的选项配置时，回退到全局属性选项
- 属性单位判断：与选项判断逻辑一致，命中某一层则使用该层的可用单位配置

**合并示例**：

以"手提包 × 爱马仕 × Birkin"创建 SPU 为例，假设三层作用域配置如下：

| 属性 | CATEGORY（手提包） | CATEGORY_BRAND（手提包×爱马仕） | CATEGORY_SERIES（手提包×爱马仕×Birkin） |
|------|-------------------|-------------------------------|---------------------------------------|
| 颜色 | 启用，必填，选项：黑色/白色/红色/棕色/金棕 | 启用，必填，选项：黑色/棕色/金棕 | 启用，必填，选项：黑色/红色/米色 |
| 材质 | 启用，必填，选项：牛皮/羊皮/鳄鱼皮 | 启用，必填，选项：Togo牛皮/鳄鱼皮 | 未配置 |
| 缝线工艺 | 未配置 | 启用，选填，选项：手工缝线/Saddle Stitch | 启用，必填，选项：Saddle Stitch |
| 尺寸 | 启用，必填，单位：mm/cm/m/inch | 未配置 | 未配置 |

最终生效结果（SPU 创建时加载的属性表单）：

| 属性 | 命中层级 | 必填 | 可用选项/单位 | 解析说明 |
|------|---------|------|-------------|---------|
| 颜色 | CATEGORY_SERIES | 必填 | 黑色/红色/米色 | 系列层有配置 → 直接用系列层，不合并上级 |
| 材质 | CATEGORY_BRAND | 必填 | Togo牛皮/鳄鱼皮 | 系列层未配置 → 向下找到品牌层 |
| 缝线工艺 | CATEGORY_SERIES | 必填 | Saddle Stitch | 系列层有配置 → 用系列层（注意必填性也被系列层覆盖为必填） |
| 尺寸 | CATEGORY | 必填 | mm/cm/m/inch | 系列层和品牌层都未配置 → 回退到类目层 |

关键要点：每个属性独立解析，逐属性按优先级查找，命中即停止，不跨层合并选项。

#### 2.8.1 属性作用域列表

**功能描述**

**页面元素**

### 2.9 SPU 管理

#### 模块概述

SPU（标准产品单元）是标品层的核心实体，代表一个标准商品（如 Hermes Birkin 25），不区分具体规格。SPU 归属于一个后台类目和一个系列，通过系列间接关联品牌。SPU 下可创建多个标品 SKU。

SPU 创建时，系统根据所选类目/品牌/系列的属性作用域规则，动态生成属性填写表单。

#### 2.9.1 SPU 列表

**功能描述**

展示所有 SPU，支持搜索和筛选，便于运营人员快速定位标品。

**页面元素**

| 元素 | 说明 |
|------|------|
| 搜索栏 | 支持按 SPU ID / SPU 名称 / 编码搜索 |
| 筛选 | 按类目、按系列筛选 |
| 列表字段 | SPU ID、SPU 名称、SPU 编码、所属类目、所属系列、SKU 数、状态、操作 |
| 操作按钮 | 新增 SPU、编辑 |

**交互说明**

- 点击"新增 SPU"跳转 SPU 新建页
- 点击"编辑"跳转 SPU 编辑页

**UI 关联**

- 原型：looply-商品管理后台原型-v28-风格A-活力亲和.html → SPU 管理（page-spu-list）

| 字段 | 类型 | 必填 | 说明 |

| 所属系列 | 下拉选择 | 是 | 选择品牌后，可选范围限定为该品牌下覆盖当前类目的系列 |

销售属性（值域选择）区域：

选择类目/品牌/系列后，系统根据属性作用域规则加载当前生效的销售属性列表。运营人员为每个销售属性选择本 SPU 启用的值域子集。

系统根据属性作用域规则加载当前生效的描述属性列表，运营人员逐个填写属性值。

| 列 | 说明 |

| 操作 | 删除 |

| 字段 | 规则 | 校验时机 |

**操作流程**

3. 系统根据属性作用域加载销售属性和描述属性
4. 为销售属性选择本 SPU 启用的值域

8. 编辑 SKU 的编码、货号、参考价
9. 点击"保存"

**异常处理**

- SPU 编码重复：提示"该 SPU 编码已存在"
- 品牌未覆盖所选类目：提示"所选品牌未覆盖该类目"
- 系列不属于所选品牌或未覆盖所选类目：提示"所选系列不匹配"
- 必填属性未填写：提示"请填写必填属性：{属性名称}"

### 2.13 实物商品管理

#### 模块概述

实物商品（product）是商品实物层的核心实体，采用一物一码模式——每件实物有唯一的内部条码，绑定一个标品 SKU。实物商品记录实物的来源、成色、质检明细、鉴定信息、价格和生命周期状态。

商品录入采用 4 步流程：关联标品 → 实物录入 → 质检鉴定 → 定价确认。

#### 状态设计原则

实物商品状态（item_status）管理实物的完整生命周期，从质检录入到最终退出经营的全过程。状态设计遵循以下原则：

1. **生命周期完整性**：状态覆盖实物从进入系统到退出的完整流程，不仅仅是"有效/无效"的粗粒度划分
2. **类型统一性**：二手商品和标准配件共用同一套状态骨架，仅在终态上有差异
3. **职责清晰性**：item_status 管理实物生命周期，listing_status 管理渠道销售，inventory 管理库存数量，三层职责分离

#### 状态枚举定义

| 状态值 | 状态名称 | 适用商品类型 | 说明 |
|--------|---------|-------------|------|
| pending_check | 待核验 | 二手商品、标准配件 | 已进入实物商品管理范围，等待质检/验收录入 |
| checking | 核验中 | 二手商品、标准配件 | 正在进行质检、验收、资料补录或人工复核 |
| pending_sale | 待销售 | 二手商品、标准配件 | 核验通过，具备销售条件，但当前未处于销售承载中 |
| on_sale | 销售中 | 二手商品、标准配件 | 已进入销售阶段，当前至少存在一个有效销售承载 |
| sold | 已售出 | 二手商品 | 销售闭环完成，不再作为可售实物流转（二手商品终态） |
| discontinued | 已停售 | 标准配件 | 该配件已退出销售经营，不再继续销售（标准配件终态） |
| invalid | 已失效 | 二手商品、标准配件 | 因质检失败、损坏、报废、丢失、合规问题等退出流转（异常终态） |

#### 状态流转规则

**二手商品状态流转**：

```
待核验 → 核验中 → 待销售 ⇄ 销售中 → 已售出
                    ↓         ↓
                 已失效    已失效

核验中 ────────→ 已失效
```

| 序号 | 起始状态 | 目标状态 | 触发条件 |
|------|---------|---------|---------|
| 1 | 待核验 | 核验中 | 开始质检/录入 |
| 2 | 核验中 | 待销售 | 质检通过/信息完整 |
| 3 | 核验中 | 已失效 | 质检不通过/报废/真伪异常 |
| 4 | 待销售 | 销售中 | 创建有效销售承载/进入在售 |
| 5 | 待销售 | 已失效 | 确认无法继续经营 |
| 6 | 销售中 | 已售出 | 成交完成 |
| 7 | 销售中 | 待销售 | 全部停售但仍保留后续销售资格 |
| 8 | 销售中 | 已失效 | 损坏/丢失/异常终止 |

**标准配件状态流转**：

```
待核验 → 核验中 → 待销售 ⇄ 销售中 → 已停售
                    ↓         ↓
                 已失效    已失效

核验中 ────────→ 已失效
```

| 序号 | 起始状态 | 目标状态 | 触发条件 |
|------|---------|---------|---------|
| 1 | 待核验 | 核验中 | 开始验收/审核 |
| 2 | 核验中 | 待销售 | 验收通过/资料通过 |
| 3 | 核验中 | 已失效 | 验收失败/不合规/作废 |
| 4 | 待销售 | 销售中 | 上架销售/进入售卖阶段 |
| 5 | 待销售 | 已停售 | 确认不再经营 |
| 6 | 销售中 | 待销售 | 全部停售但后续还会继续卖 |
| 7 | 销售中 | 已停售 | 业务决策永久停销 |
| 8 | 销售中 | 已失效 | 损坏/作废/异常退出 |

#### 状态联动规则

**实物商品状态与渠道商品状态联动**：

| 触发条件 | 联动动作 | 说明 |
|---------|---------|------|
| item_status 从销售中变为非销售中 | 关联的所有 active listing 自动转为 off_shelf | 实物不可用时，所有在售渠道自动下架 |
| item_status 从非销售中恢复为待销售 | listing 不自动恢复 | 需运营手动重新上架 |
| item_status = 已售出 | 禁止创建新 listing | 已售出商品不可再上架 |
| item_status = 已失效 | 禁止创建新 listing | 已失效商品不可再上架 |
| item_status = 已停售 | 禁止创建新 listing | 已停售配件不可再上架 |

**可卖判断规则**：

```
可卖 = listing_status = active AND available_qty > 0
```

item_status 不直接参与实时可卖判断，通过联动规则约束 listing 的创建和维持。

#### 派生展示状态

为便于商户和平台运营人员理解，系统提供派生展示状态，根据 item_status、listing_status 和 available_qty 计算得出：

**标准配件展示状态**：

| item_status | listing 情况 | available_qty | 展示状态 |
|------------|-------------|--------------|---------|
| 销售中 | 存在 active listing | > 0 | 在售 |
| 销售中 | 存在 active listing | = 0 | 售罄 |
| 待销售 | - | - | 待销售 |
| 核验中 | - | - | 核验中 |
| 已停售 | - | - | 已停售 |
| 已失效 | - | - | 已失效 |

**二手商品展示状态**：

| item_status | listing 情况 | 展示状态 |
|------------|-------------|---------|
| 销售中 | 存在 active listing | 在售 |
| 待销售 | - | 待销售 |
| 核验中 | - | 核验中 |
| 已售出 | - | 已售出 |
| 已失效 | - | 已失效 |

### 2.14 渠道商品管理

#### 模块概述

渠道商品（listing）记录一件实物商品在某个渠道的上架信息。同一件实物可在多个渠道同时上架，每个渠道有独立的挂牌价和上架状态。渠道商品通过商品列表的"快捷上架"操作创建。

#### 状态枚举定义

| 状态值 | 状态名称 | 说明 |
|--------|---------|------|
| active | 在售 | 正在该渠道销售 |
| off_shelf | 已下架 | 主动撤卖或被动下架 |

创建渠道商品（上架操作）后初始状态为 active（在售）。

**扩展字段**：

| 字段 | 类型 | 说明 |
|------|------|------|
| off_shelf_reason | ENUM | 下架原因：manual（主动下架）/ out_of_stock（缺货下架）/ blocked（违规拦截）/ item_invalid（实物状态联动下架） |
| off_shelf_remark | VARCHAR | 下架原因详情（如拦截原因、违规说明等） |
| is_deleted | BOOLEAN | 软删除标记，默认 false |

#### 状态流转规则

```
在售 ⇄ 已下架
```

| 序号 | 起始状态 | 目标状态 | 触发条件 |
|------|---------|---------|---------|
| 1 | 在售 | 已下架 | 主动撤卖/缺货下架/违规拦截/实物状态联动 |
| 2 | 已下架 | 在售 | 重新上架（需实物状态为待销售或销售中） |

### 2.15 渠道管理

#### 模块概述

渠道是二手商品的销售通路，支持三种类型：线上自营（官网商城）、线下门店、三方平台（如亚马逊）。每个渠道有独立的运营配置。

#### 渠道命名规范

渠道名称遵循统一的命名规范：**站点-类型-自定义内容**

- **站点**：国家名称（如美国、英国、德国、加拿大、澳大利亚），从国家枚举中选择
- **类型**：渠道类型（线上、线下、三方平台）
- **自定义内容**：具体渠道名称（如官网商城、旗舰店、亚马逊）

**命名示例**：
- 美国-线上-官网商城
- 德国-线下-旗舰店
- 美国-三方平台-亚马逊

**设计目的**：
1. 统一命名格式，便于系统化管理
2. 站点信息前置，支持多国站点扩展
3. 类型明确，便于按渠道类型筛选和统计

## 三、依赖与风险

### 上下游系统依赖

本次覆盖的模块为商品系统全量 MVP 功能。外部系统依赖如下：

| 外部系统 | 依赖说明 |
|---------|---------|
| 商家系统 | 实物商品列表和渠道商品列表展示卖家 ID 和卖家昵称，需从商家系统读取 |
| 图片存储服务 | SPU 图片、SKU 图片、实物图片的上传和存储 |

### 下游影响

| 下游模块 | 依赖说明 |
|---------|---------|
| 搜索系统 | 商品上架/下架时需通知搜索系统更新索引 |
| 订单系统 | 下单时需校验商品可售性（库存状态 + 上架状态） |
| 支付结算系统 | 读取商品售价用于订单结算 |

### 风险项

| 风险 | 影响 | 应对 |
|------|------|------|
| 属性作用域规则复杂度高 | 三层优先级叠加可能导致运营配置困难 | 提供"生效预览"功能，让运营人员在保存前确认最终效果 |
| 度量单位基准切换 | 切换基准单位影响已存储的属性值 | 切换时强制确认，并提示影响范围 |
| 一物多渠道库存一致性 | 同一实物在多渠道上架，售出后需全渠道同步下架 | 库存状态改为"已售"时自动下架所有渠道 |
| 商家系统对接 | 商品列表和渠道商品列表需展示卖家信息 | 需确认商家系统接口可用性和字段映射 |
